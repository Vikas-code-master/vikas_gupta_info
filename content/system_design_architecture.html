Golden rule of scale: the closer a layer is to the user, the cheaper it is to absorb traffic there. Push reads to the edge, push writes to async queues, and protect the database with caches.

2. Client Layer

Browsers, mobile apps, third-party API consumers, and IoT devices.

How it scales

  • HTTP/2 & HTTP/3 (QUIC): multiplexing reduces connection setup cost — critical at millions of concurrent users.
  • Client-side caching: ETag, Cache-Control, IndexedDB, service workers — avoid hitting the network at all.
  • Exponential backoff & jitter on retries to prevent thundering-herd after an outage.
  • Connection reuse / keep-alive drastically cuts TCP+TLS handshakes.
Impact at 1M RPS: a 200ms cache TTL on the client side can reduce backend hits by 30–50% on hot endpoints.

3. DNS & Anycast Routing

The first stop. DNS resolves api.example.com to an IP — and at scale, it does much more than that.

Capabilities

  • GeoDNS: route Indian users to ap-south-1, US users to us-east-1.
  • Anycast IPs: the same IP is announced from many POPs; BGP routes the user to the nearest one.
  • Latency-based routing & health checks for automatic failover.

How it handles millions

DNS is hierarchical and heavily cached at OS, browser, and resolver levels. Providers like Route 53 / Cloudflare DNS handle trillions of queries/day because resolution is stateless and the TTL (e.g. 60s) means most queries never reach the authoritative servers.

4. CDN — Content Delivery Network

Geographically distributed edge servers cache static and (increasingly) dynamic content close to the user.

What lives at the edge

  • Static assets: JS, CSS, images, video segments (HLS/DASH).
  • API responses with public/cacheable headers.
  • Edge functions / workers (Cloudflare Workers, Lambda@Edge) for personalization and A/B routing.

How it handles millions

  • A typical CDN absorbs 70–95% of total traffic for a content-heavy site.
  • Hierarchical caching: edge POP → regional shield → origin, so even a cache miss rarely hits the origin.
  • TLS termination and DDoS scrubbing happen at the edge — origin sees a clean stream.
Cache hit ratio is the single most important CDN metric. Target ≥ 90% for media; ≥ 60% for cacheable APIs.

5. Load Balancer

Distributes incoming requests across many backend servers so no single instance is overwhelmed.

Where the Application Load Balancer (ALB / L7 LB) sits

An Application Load Balancer is an L7 (HTTP-aware) load balancer. In the request path it sits after the CDN and the L4 / Network Load Balancer, and just before the API Gateway and the application services. It is the first component in the stack that actually parses HTTP — so it can route by host, path, header, or cookie.

Client  ──▶  DNS  ──▶  CDN  ──▶  L4 / Network LB (TCP, TLS termination)
                                          │
                                          ▼
                          ┌──────────────────────────────┐
                          │  Application Load Balancer   │  ◀── you are here
                          │  (AWS ALB / NGINX / Envoy /  │      • L7 / HTTP
                          │   GCP HTTPS LB / Azure AGW)  │      • path & host routing
                          └───────────────┬──────────────┘      • WAF, sticky sessions
                                          ▼
                                  API Gateway
                                          ▼
                          Microservices (Orders, Users, …)

In smaller deployments the L4 LB is often skipped and the ALB is exposed directly to the internet (behind the CDN). In a Kubernetes cluster, the ALB usually maps to an Ingress controller (AWS ALB Ingress, NGINX Ingress, Istio Gateway) sitting at the cluster edge — the public entry point for every service inside the cluster.

Types

LayerOperates onExamplesUse
L4 (Transport)TCP/UDP, IP+portAWS NLB, IPVSUltra-fast, millions of connections, no payload inspection
L7 (Application)HTTP/gRPC headers, pathsNGINX, Envoy, AWS ALB, HAProxyPath/host routing, sticky sessions, header rewriting
Global (DNS-based)Region selectionAWS Route 53, GCP GLBMulti-region failover

Algorithms

  • Round Robin — simple, ignores server load.
  • Least Connections — best for long-lived connections.
  • Weighted — heterogeneous instance sizes.
  • Consistent Hashing — for sticky cache routing (e.g. by user_id).
  • EWMA / Power-of-Two-Choices — modern, picks 2 random servers and sends to the less-loaded.

How it handles millions

  • Hardware-offloaded LBs (AWS NLB) handle tens of millions of TCP flows per LB.
  • Health checks remove unhealthy nodes in seconds → graceful degradation.
  • Auto-scales horizontally; behind a global anycast IP it appears as a single endpoint.
  • SSL termination offloads CPU-heavy crypto from app servers.
Pattern: always run the LB in at least two AZs; a single LB is a single point of failure.

ALB vs NLB — in plain language

Both spread traffic across backend servers. The difference is how deeply they look inside the traffic.

NLB — Network Load Balancer (Layer 4)

A doorman who only checks your ticket number, not what's inside your bag.

  • Works at TCP / UDP level — sees IPs and ports, nothing else.
  • Does not read the HTTP request, headers, URL, or cookies.
  • Just forwards raw packets to a backend server.
  • Insanely fast and cheap — almost no work per packet.

Use when: raw speed (millions of conn/sec), non-HTTP traffic (game servers, MQTT, databases), need a static IP for whitelisting.

Analogy: a highway toll booth — just lets cars through, doesn't care where they're going.

ALB — Application Load Balancer (Layer 7)

A smart receptionist who reads your form and sends you to the right department.

  • Works at HTTP / HTTPS level — actually parses the request.
  • Routes by URL path (/orders/*), hostname, header, cookie, query string.
  • Handles HTTPS termination, sticky sessions, WebSockets, redirects.
  • Slower than NLB (parsing costs CPU) but much smarter.

Use when: web apps, REST APIs, microservices needing path-based routing, sticky sessions, or WAF integration.

Analogy: hospital reception — reads your symptoms and sends you to Cardiology vs Orthopedics.

Side-by-side

FeatureNLB (Layer 4)ALB (Layer 7)

ProtocolsTCP, UDP, TLSHTTP, HTTPS, WebSocket, gRPC

One-line summary: NLB = dumb but fast packet forwarder. ALB = smart HTTP-aware traffic router. In big systems you often use both — NLB at the very edge (firehose + static IP), then forwarding to ALBs that do the smart routing into microservices.

6. API Gateway

The single entry point for all client API traffic to your microservice fleet. It is not the same thing as a load balancer — though it sits next to one.

Responsibilities

  • Routing & composition — map /orders/* to the orders service, fan-out to multiple services and aggregate.
  • Authentication / Authorization — JWT / OAuth2 / API-key validation before traffic hits services.
  • Rate limiting & quotas — token bucket / leaky bucket per user / API key / IP.
  • Request / response transformation — protocol bridging (REST ↔ gRPC, GraphQL).
  • Caching of idempotent GETs.
  • Circuit breaking, retries, timeouts centralized.
  • Schema validation & WAF integration.

Examples

Kong, AWS API Gateway, Apigee, Tyk, Envoy + Istio, Spring Cloud Gateway.

How it handles millions

  • Stateless & horizontally scalable behind the L7 LB — add more pods as RPS grows.
  • Rate limiters use a fast shared store (Redis) with token-bucket counters; each check is O(1).
  • Centralizing cross-cutting concerns means each microservice becomes simpler and faster.
  • Connection pooling to upstream services keeps tail-latency predictable.
Trap: a fat gateway becomes a SPOF. Keep custom logic minimal; push business logic into services.

How a "single" API Gateway handles all traffic from the ALB

The "single API Gateway" is a logical entry point, not a single machine. Physically it's a horizontally-scaled fleet of stateless gateway instances sitting behind the ALB.

                          ALB (L7)
                            │
          ┌─────────────────┼─────────────────┐
          ▼                 ▼                 ▼
     gateway-pod-1    gateway-pod-2  …  gateway-pod-N
  1. ALB fans out to many gateway instances. Round-robin / least-conn picks one pod per request. A single Kong / Envoy / AWS API Gateway pod handles 20k–100k RPS; 50 pods → millions of RPS.
  2. Gateway instances are stateless. No session, no in-memory user data. Any pod can serve any request — that is why you can have N of them behind one ALB.
  3. Shared state lives in fast external stores.
    • Rate-limit counters → Redis (atomic INCR with TTL, O(1) per check).
    • JWT validation → stateless (verify signature with cached public keys).
    • Auth / session lookups → Redis cache; DB only on miss.
    • Routing config → loaded at boot, refreshed via control plane (e.g. xDS for Envoy).
  4. Async, non-blocking I/O. Gateways like Envoy / NGINX / Kong use event loops — one pod can hold tens of thousands of in-flight requests because each is just a file descriptor, not a thread.
  5. Connection pooling to upstreams. Gateway holds a small pool of HTTP/2 / gRPC connections per downstream service, multiplexing many requests onto each. 100k incoming RPS does not mean 100k outgoing TCP connections.
  6. Auto-scaling on the right signal. HPA scales replicas on RPS, p95 latency, or active connections — not just CPU. New pods register with discovery in 10–30s.
  7. The ALB itself is also distributed. AWS ALB / GCP HTTPS LB are multi-AZ, auto-scaled fleets behind a single anycast IP — never a bottleneck.
Mental model: "API Gateway" = a logical role. Capacity = replicas × per-pod RPS. The only true singletons are the config (routes, policies) and the external state (Redis, auth provider) — and both are themselves clustered.

Multiple API Gateways — when one isn't enough

In real systems you almost always have multiple gateways, in two different senses.

1. Multiple instances of the same gateway (horizontal scaling)

One logical gateway, many pods behind the ALB. Same config, same routes, just more copies for capacity and HA — exactly what was described above.

2. Multiple different gateways (specialized per audience — the BFF pattern)

Different client types have very different needs, so big systems run separate gateways for each. This is the Backend-for-Frontend (BFF) pattern.

   Mobile App        Web App         Partner B2B       Internal Admin

  │ Gateway │    │ Gateway │       │  Gateway    │   │  Gateway    │
  └────┬────┘    └────┬────┘       └──────┬──────┘   └──────┬──────┘
       │              │                   │                  │
       └──────────────┴────────┬──────────┴──────────────────┘
                               ▼
                    Microservices fleet
                (Orders, Users, Payments, …)
GatewayAudienceWhy it's separate
Mobile GatewayiOS / Android appsSlim payloads, aggregates many calls into one (saves battery + data); REST or GraphQL

Each gateway has its own config, own auth rules, own rate limits, own scaling profile — but they all talk to the same underlying microservices.

How traffic gets routed to the right gateway

Usually by hostname at the ALB:

api.example.com         → Public / Partner Gateway
m.api.example.com       → Mobile Gateway
app.example.com/api/*   → Web Gateway
admin.example.com       → Internal Gateway

Why not just one giant gateway for everyone?

  • Blast radius: a bad config push for partner APIs shouldn't break the mobile app.
  • Different scaling needs: mobile traffic spikes at 9 AM; partner APIs are steady all day. Scale them independently.
  • Different security postures: partner gateway needs strict mTLS + WAF; internal gateway just needs SSO.
  • Different teams own them: mobile team owns mobile gateway, platform team owns partner gateway.
  • Independent deploys: ship a change to one without redeploying the others.
Summary: A "single API Gateway" is a logical role, not a single box. In practice you have N instances of M different gateways, each tailored to a specific audience, all fronting the same shared microservices.

7. Reverse & Forward Proxies

Reverse Proxy

Sits in front of servers; clients don't know which backend served them.

  • TLS termination, gzip/brotli compression.
  • Static-file serving, caching, request buffering.
  • Examples: NGINX, Envoy, HAProxy, Caddy.
  • Often combined with the L7 load balancer role.

Forward Proxy

Sits in front of clients; servers don't know who the real client is.

  • Egress control from corporate networks.
  • Outbound rate-limiting to third-party APIs.
  • Examples: Squid, corporate egress gateway, sidecar Envoy.

When you need a proxy beyond an LB

  • Service Mesh sidecar (Envoy/Linkerd): mTLS, retries, traffic shifting between every pair of services. Essential at 100s of microservices.
  • Egress proxy for outbound calls to flaky third-party APIs (payment gateways, SMS providers) — adds retries and circuit breakers without polluting service code.

How it handles millions

A single Envoy/NGINX instance can sustain 50k–200k RPS on commodity hardware with keep-alive. Add instances horizontally; mesh sidecars scale linearly with pods.

8. Application / Service Layer

Where business logic lives — typically as microservices, each owning a bounded context.

Design principles

  • Stateless services → any instance can serve any request → trivial horizontal scale.
  • Single responsibility per service (Auth, Orders, Inventory, Notifications…).
  • Database per service — no shared DB between services; cross-service data via APIs or events.
  • Idempotent endpoints with idempotency keys → safe retries.
  • Graceful degradation — feature flags, fallbacks (e.g. show stale data if recommendations fail).

Runtime

Containers on Kubernetes (or ECS / Nomad). Horizontal Pod Autoscaler scales on CPU, memory, or custom metrics (e.g. queue depth, RPS). Cluster Autoscaler adds nodes as pods can't be scheduled.

Rule of thumb: if you can't kill any single pod without an alert firing, your service isn't horizontally scalable yet.

9. Synchronous Communication

Caller waits for a response. Used when the client genuinely needs the result before continuing.

Protocols

ProtocolWhen to useProsCons
REST / HTTP/JSONPublic APIs, browsersUniversal, cacheable, debuggableVerbose, no schema enforcement
gRPC (HTTP/2 + Protobuf)Internal service-to-serviceFast binary, streaming, schemaBrowser support needs grpc-web
GraphQLClient-driven aggregationOne round trip, no over-fetchN+1, caching complexity
WebSockets / SSEReal-time pushLow latency, bidirectionalStateful — sticky LB needed

Resilience patterns (non-negotiable at scale)

  • Timeouts: every call must have one — never wait forever.
  • Retries with exponential backoff + jitter on idempotent operations only.
  • Circuit breaker (Hystrix / Resilience4j / Envoy): trip after N failures, fail fast for a cool-off period.
  • Bulkhead: dedicated thread/connection pools per dependency so one slow service can't drown the rest.
  • Hedged requests: send a duplicate after p99 to cut tail latency.

How it handles millions

Connection pooling + HTTP/2 multiplexing keeps connection counts manageable. Service mesh sidecars enforce timeouts/retries uniformly. But synchronous chains amplify failures — keep them short (≤ 3 hops).

9.1 Deep Dive: gRPC internal RPC

gRPC is an open-source, high-performance RPC framework released by Google in 2015 and now governed by the CNCF. It is the de-facto choice for east–west (service-to-service) communication in modern microservice stacks. The "g" originally stood for Google; today it is reinterpreted differently in every release.

Why gRPC instead of REST/JSON?

PropertygRPCREST + JSON
TransportHTTP/2 (multiplexed, header-compressed)HTTP/1.1 (typically)
PayloadProtocol Buffers (binary, 3–10× smaller)JSON text
Schema / ContractMandatory .proto file — strongly typed, code-generatedOptional (OpenAPI), hand-written DTOs
StreamingFirst-class: server, client & bidirectional streamsSSE / WebSockets bolted on
Browser supportNeeds grpc-web proxyNative
Latency at p99Lower (binary + HTTP/2 + persistent connections)Higher
ToolingStubs in 10+ languages from one .protoHand-write or generate from OpenAPI

The 4 RPC types

  • Unary — classic request/response. rpc GetUser(GetUserRequest) returns (User);
  • Server streaming — client sends one request, server replies with a stream. Good for paginated results, live tickers, log tails.
  • Client streaming — client uploads a stream, server returns one response. Good for chunked uploads, telemetry batches.
  • Bidirectional streaming — both sides stream independently over a single HTTP/2 connection. Good for chat, real-time gaming state, mutual telemetry.

1. Define the contract — user.proto

syntax = "proto3";

package com.example.user;

option java_multiple_files = true;
option java_package = "com.example.user.grpc";
option java_outer_classname = "UserProto";

service UserService {
  // Unary
  rpc GetUser (GetUserRequest) returns (User);

  // Server streaming — emit one User per matching record
  rpc SearchUsers (SearchRequest) returns (stream User);

  // Client streaming — bulk upload then summary
  rpc BulkCreate (stream User) returns (BulkCreateResponse);

  // Bidirectional — live presence channel
  rpc Presence (stream PresenceEvent) returns (stream PresenceEvent);
}

message GetUserRequest { string user_id = 1; }

message User {
  string user_id = 1;
  string email   = 2;
  string name    = 3;
  int64  created_at_epoch_ms = 4;
}

message SearchRequest { string query = 1; int32 limit = 2; }

message BulkCreateResponse { int32 created = 1; int32 failed = 2; }

message PresenceEvent {
  string user_id = 1;
  enum Status { OFFLINE = 0; ONLINE = 1; AWAY = 2; }
  Status status = 2;
}

The Protobuf compiler (protoc) + the gRPC Java plugin turn this .proto into Java classes: request/response POJOs, a server abstract base class (UserServiceGrpc.UserServiceImplBase), and three client stubs (blocking, async, future).

3. Server implementation (Java)

package com.example.user.grpc;

import io.grpc.Server;
import io.grpc.ServerBuilder;
import io.grpc.Status;
import io.grpc.stub.StreamObserver;

import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;

public class UserServer {

  // In-memory store for illustration
  private static final Map<String, User> STORE = new ConcurrentHashMap<>();

  static class UserServiceImpl extends UserServiceGrpc.UserServiceImplBase {

    // -------- Unary --------
    @Override
    public void getUser(GetUserRequest req, StreamObserver<User> resp) {
      User u = STORE.get(req.getUserId());
      if (u == null) {
        resp.onError(Status.NOT_FOUND
            .withDescription("user " + req.getUserId() + " not found")
            .asRuntimeException());
        return;
      }
      resp.onNext(u);
      resp.onCompleted();
    }

    // -------- Server streaming --------
    @Override
    public void searchUsers(SearchRequest req, StreamObserver<User> resp) {
      int sent = 0;
      for (User u : STORE.values()) {
        if (u.getName().toLowerCase().contains(req.getQuery().toLowerCase())) {
          resp.onNext(u);                  // push each match
          if (++sent >= req.getLimit()) break;
        }
      }
      resp.onCompleted();
    }

    // -------- Client streaming --------
    @Override
    public StreamObserver<User> bulkCreate(StreamObserver<BulkCreateResponse> resp) {
      return new StreamObserver<>() {
        int ok = 0, bad = 0;
        @Override public void onNext(User u) {
          if (u.getUserId().isEmpty()) { bad++; return; }
          STORE.put(u.getUserId(), u);
          ok++;
        }
        @Override public void onError(Throwable t) { /* log */ }
        @Override public void onCompleted() {
          resp.onNext(BulkCreateResponse.newBuilder()
              .setCreated(ok).setFailed(bad).build());
          resp.onCompleted();
        }
      };
    }

    // -------- Bidirectional streaming --------
    @Override
    public StreamObserver<PresenceEvent> presence(StreamObserver<PresenceEvent> resp) {
      return new StreamObserver<>() {
        @Override public void onNext(PresenceEvent ev) {
          // echo back to all subscribers in real code — simplified here
          resp.onNext(ev);
        }
        @Override public void onError(Throwable t) { resp.onError(t); }
        @Override public void onCompleted()        { resp.onCompleted(); }
      };
    }
  }

  public static void main(String[] args) throws Exception {
    Server server = ServerBuilder.forPort(9090)
        .addService(new UserServiceImpl())
        .build()
        .start();
    System.out.println("gRPC UserService listening on 9090");
    Runtime.getRuntime().addShutdownHook(
        new Thread(server::shutdown));
    server.awaitTermination();
  }
}

4. Client — three stub flavors

package com.example.user.grpc;

import io.grpc.ManagedChannel;
import io.grpc.ManagedChannelBuilder;
import io.grpc.stub.StreamObserver;

import java.util.Iterator;
import java.util.concurrent.TimeUnit;

public class UserClient {

  public static void main(String[] args) throws Exception {
    ManagedChannel channel = ManagedChannelBuilder
        .forAddress("users.svc", 9090)
        .usePlaintext()           // TLS turned off for local demo
        .keepAliveTime(30, TimeUnit.SECONDS)
        .build();

    // -------- Blocking stub (unary) --------
    var blocking = UserServiceGrpc.newBlockingStub(channel)
        .withDeadlineAfter(200, TimeUnit.MILLISECONDS);   // per-call timeout

    User u = blocking.getUser(GetUserRequest.newBuilder()
        .setUserId("u-42").build());
    System.out.println("Got: " + u.getEmail());

    // -------- Blocking stub (server streaming) --------
    Iterator<User> it = blocking.searchUsers(
        SearchRequest.newBuilder().setQuery("ali").setLimit(50).build());
    while (it.hasNext()) System.out.println(it.next().getName());

    // -------- Async stub (client streaming) --------
    var async = UserServiceGrpc.newStub(channel);
    StreamObserver<User> sink = async.bulkCreate(new StreamObserver<>() {
      @Override public void onNext(BulkCreateResponse r)  {
        System.out.println("created=" + r.getCreated());
      }
      @Override public void onError(Throwable t)          { t.printStackTrace(); }
      @Override public void onCompleted()                 { /* done */ }
    });
    for (int i = 0; i < 1000; i++) {
      sink.onNext(User.newBuilder()
          .setUserId("u-" + i).setName("user " + i).build());
    }
    sink.onCompleted();

    channel.shutdown().awaitTermination(5, TimeUnit.SECONDS);
  }
}
Stub flavors: Blocking — simplest, OK for unary & server-streaming. Async — callback-based, required for client/bidirectional streaming. Future — returns ListenableFuture, handy when composing several calls in parallel without going fully reactive.

5. Deadlines & cancellation

Every call should carry a deadline — an absolute wall-clock instant by which the response must arrive. Deadlines propagate automatically across hops (User → Order → Inventory), so if the client gives up, every downstream call is canceled too. This is the single most effective defense against cascading failures. 

// Deadline on a single call
blocking.withDeadlineAfter(150, TimeUnit.MILLISECONDS)
        .getUser(req);

// Deadline on every call made through this stub
var fastStub = UserServiceGrpc.newBlockingStub(channel)
        .withDeadlineAfter(150, TimeUnit.MILLISECONDS);

6. Interceptors — auth, logging, tracing

Interceptors are gRPC's middleware. They run for every RPC and are the right place to attach JWTs, emit metrics, or propagate trace IDs.

// --- Client-side: attach a bearer token to every call ---
public class AuthInterceptor implements ClientInterceptor {
  private final String token;
  public AuthInterceptor(String token) { this.token = token; }

  @Override
  public <Req, Res> ClientCall<Req, Res> interceptCall(
      MethodDescriptor<Req, Res> method, CallOptions opts, Channel next) {
    return new ForwardingClientCall.SimpleForwardingClientCall<>(
        next.newCall(method, opts)) {
      @Override public void start(Listener<Res> r, Metadata headers) {
        headers.put(Metadata.Key.of("authorization",
            Metadata.ASCII_STRING_MARSHALLER), "Bearer " + token);
        super.start(r, headers);
      }
    };
  }
}

ManagedChannel secured = ManagedChannelBuilder
    .forAddress("users.svc", 9090)
    .useTransportSecurity()
    .intercept(new AuthInterceptor(jwt))
    .build();

// --- Server-side: validate the same header ---
public class AuthServerInterceptor implements ServerInterceptor {
  @Override
  public <Req, Res> ServerCall.Listener<Req> interceptCall(
      ServerCall<Req, Res> call, Metadata headers,
      ServerCallHandler<Req, Res> next) {
    String auth = headers.get(Metadata.Key.of(
        "authorization", Metadata.ASCII_STRING_MARSHALLER));
    if (auth == null || !auth.startsWith("Bearer ")) {
      call.close(Status.UNAUTHENTICATED.withDescription("missing token"),
                 new Metadata());
      return new ServerCall.Listener<>() {};
    }
    return next.startCall(call, headers);
  }
}

8. Built-in load balancing & service discovery

Unlike REST, where the client opens a fresh HTTP/1.1 connection per request, gRPC keeps a persistent HTTP/2 connection — which would normally pin a client to one backend. gRPC solves this with name resolvers and load-balancing policies baked into the client. 

ManagedChannel channel = ManagedChannelBuilder
    .forTarget("dns:///users.svc.cluster.local:9090")  // resolves all A records
    .defaultLoadBalancingPolicy("round_robin")          // built-in policy
    .build();

In Kubernetes the dns:/// resolver picks up every pod IP behind a headless Service, so requests are spread across replicas without an external L7 LB. For richer policies (locality-aware, weighted, outlier ejection) plug in an Envoy / Istio sidecar and let it do xDS-driven LB.

10. Practical scaling notes

  • One channel per upstream, not per call. ManagedChannel is thread-safe and multiplexes thousands of in-flight RPCs over a single HTTP/2 connection.
  • Watch the 4 MB default message size. Override with .maxInboundMessageSize(N) if you legitimately need bigger — otherwise stream.
  • Enable keepalives (keepAliveTime / keepAliveTimeout) so dead connections get detected before they swallow requests.
  • Always set a deadline — Java gRPC will not time out on its own.
  • Use server reflection + grpcurl in non-prod for easy debugging without writing client code.
  • For browsers / mobile-web, terminate at an grpc-web proxy (Envoy filter) or expose a REST gateway.
Rule of thumb: if both sides of a call are your own services and live inside the same trust boundary, default to gRPC. Keep REST for public APIs, third-party integrations, and browsers.

10. Asynchronous Communication

Caller hands off the work and returns immediately. The receiver processes when ready. This is how internet-scale systems absorb spikes.

Patterns

  • Message Queue (point-to-point) — RabbitMQ, AWS SQS. One consumer gets each message.
  • Pub/Sub / Event Bus — Kafka, Google Pub/Sub, NATS. Many consumers, each gets a copy.
  • Event Streaming — Kafka with retention; replayable history. Powers event sourcing & CDC.
  • Webhooks — async outbound notifications to third parties.

Why async at scale

  • Decoupling: producer doesn't know or care about consumers. Add new consumers without touching producers.
  • Buffering / load smoothing: Black Friday spike of 10× → queue absorbs it; consumers drain at their pace.
  • Resilience: if a downstream service is down, messages wait in the queue instead of being lost.
  • Parallelism: partition the topic by key (e.g. user_id), scale consumers horizontally per partition.

Critical concerns

  • Delivery semantics: at-most-once, at-least-once (most common — needs idempotent consumers), exactly-once (Kafka transactions, expensive).
  • Ordering: guaranteed only within a partition — pick the partition key carefully.
  • Dead-Letter Queue (DLQ): messages that fail after N retries land here for manual investigation.
  • Backpressure & lag monitoring: alert on consumer lag growing.
  • Outbox pattern: write to DB and outbox table in one transaction; a relay publishes to the broker — prevents lost events.

How it handles millions

Kafka commodity clusters routinely sustain millions of messages/sec. Throughput scales by adding partitions and consumer-group members. SQS scales transparently to nearly unlimited TPS.

Heuristic: if the user doesn't need the answer right now (emails, analytics, fanout, image processing, search indexing), make it async. The synchronous path stays fast and the system stays resilient.

11. Caching Layer

Caches exist at every layer — browser, CDN, gateway, application, database. Each level is 10–100× faster than the next.

Distributed cache (Redis / Memcached)

  • Redis Cluster: sharded by hash slot, replicas per shard, sub-ms latency.
  • Used for: session stores, rate-limit counters, leaderboards, hot-row caching, pub/sub.
  • Single Redis node ≈ 100k ops/sec; a 16-shard cluster ≈ 1.5M+ ops/sec.

Cache strategies

StrategyBehaviorBest for
Cache-aside (lazy)App reads cache; on miss, reads DB, writes cacheRead-heavy, tolerant of staleness
Read-throughCache itself reads from DB on missSimpler app code
Write-throughWrites go to cache + DB synchronouslyStrong consistency
Write-behindWrites to cache; flushed to DB asyncWrite-heavy, can lose writes on crash

Pitfalls

  • Thundering herd / cache stampede: on TTL expiry, many requests hit DB at once → use single-flight or probabilistic early refresh (XFetch).
  • Cache penetration: queries for non-existent keys hit DB → cache negative results or use a Bloom filter.
  • Hot keys: one celebrity's profile drowns one shard → replicate hot keys, use local in-process LRU as L1.
  • Invalidation is genuinely the hardest problem — prefer short TTLs over fancy invalidation.

12. Databases

Pick the right tool

TypeExamplesBest for
Relational (OLTP)PostgreSQL, MySQL, AuroraTransactional data, joins, ACID

Scaling techniques

  • Vertical scale (scale-up): bigger box. Easy, finite.
  • Read replicas: stream WAL/binlog to replicas; route reads to them. Beware replication lag → eventual consistency.
  • Sharding (horizontal partitioning): split data by key (range, hash, geo). Each shard is its own primary. Cross-shard queries become hard.
  • Federation: split DBs by domain (users-db, orders-db).
  • Multi-region active-active: Spanner, CockroachDB, DynamoDB Global Tables, Cassandra. Offers cross-region writes; pay in latency.
  • CQRS: separate write model (normalized) from read model (denormalized projections).

CAP & PACELC

Under a network partition, you choose Consistency or Availability. Even when there is no partition (PACELC), you trade latency vs. consistency.

  • CP: Spanner, HBase, ZooKeeper, etcd — refuse writes if quorum unreachable.
  • AP: Cassandra, DynamoDB (default), Couchbase — accept writes; reconcile later.

How a DB handles millions

  • Most reads served from cache; the DB sees only the misses + writes.
  • Connection pool (PgBouncer / RDS Proxy) → 100k clients share 200 DB connections.
  • Indexes on access paths; avoid full scans in OLTP.
  • Async commits and group commits amortize fsync cost.
  • Heavy workloads partitioned across many shards (e.g. Instagram on Postgres, Discord on ScyllaDB).
Rule: the database is your most expensive and least scalable component. Every layer above exists to protect it.

13. Search & Analytics

  • Search: Elasticsearch / OpenSearch / Solr. Inverted index, distributed across shards. Updated via CDC or Kafka pipeline.
  • Real-time analytics: Druid, Pinot, ClickHouse — sub-second aggregations over billions of rows.
  • Batch analytics / DWH: BigQuery, Snowflake, Redshift, Spark. Fed by ETL/ELT from operational DBs.
  • Stream processing: Flink, Kafka Streams, Spark Streaming for windowed aggregations and joins.

Don't run analytical queries on the OLTP database — replicate data into a system designed for it.

14. Object & Blob Storage

  • S3 / GCS / Azure Blob for images, videos, backups, ML models, logs.
  • 11-nines durability, virtually unlimited scale, pay-per-GB.
  • Serve user uploads via pre-signed URLs — clients upload directly to S3 without going through your servers, saving bandwidth and CPU.
  • Front it with a CDN for read-heavy assets.

15. Observability

You can't operate a distributed system you can't see. The three pillars:

  • Metrics — Prometheus, Datadog, CloudWatch. Numeric time-series. RED (Rate, Errors, Duration), USE (Utilization, Saturation, Errors).
  • Logs — ELK / Loki / Splunk. Structured JSON, sampled in hot paths.
  • Traces — Jaeger / Tempo / X-Ray with OpenTelemetry. Follow a single request across every service.

Plus: SLOs & error budgets, alerting on symptoms (not causes), and synthetic + real-user monitoring.

16. Security & Auth

  • Edge: WAF (Cloudflare, AWS WAF), DDoS scrubbing, bot management.
  • Transport: TLS 1.3 everywhere, mTLS between services (service mesh).
  • AuthN: OAuth 2.0 / OIDC (Auth0, Cognito, Keycloak), short-lived JWTs, refresh tokens.
  • AuthZ: RBAC, ABAC; centralized policy engines like OPA.
  • Secrets: Vault, AWS Secrets Manager, KMS — never in env files committed to git.
  • Defense in depth: rate-limit at gateway, validate at service, parameterize queries at DB. Assume any single layer can fail.

17. Capacity Math & Scaling Rules of Thumb

Quick numbers an architect should have in their head when sizing a system for "millions of users".

Translating users → RPS

1M DAU × ~20 actions/day ÷ 86,400 s ≈ 230 RPS average. Peak ≈ 5×–10× average → ~2,000 RPS. Ten million DAU → ~20k RPS peak.

Useful latency numbers

OperationApprox latency
L1 cache reference~1 ns
Main memory reference~100 ns
SSD random read~100 µs
Round trip in same DC~500 µs
Redis GET (same DC)~1 ms
DB indexed query1–10 ms
Cross-region RTT (US ↔ EU)~80–100 ms

Single-instance ballparks

  • NGINX / Envoy: ~50–100k RPS reverse-proxying.
  • Modern Java/Go service: 5–20k RPS depending on logic.
  • PostgreSQL on good hardware: 10–50k simple reads/sec; ~5–10k writes/sec single primary.
  • Redis single shard: ~100k ops/sec.
  • Kafka per broker: 100k–1M messages/sec depending on size and acks.

Scaling checklist

  1. Make services stateless; push state to caches/DBs.
  2. Cache aggressively at every layer — but invalidate predictably.
  3. Move non-critical work to async queues.
  4. Shard the database before it becomes urgent.
  5. Eliminate SPOFs: every component multi-AZ, every dependency has a circuit breaker.
  6. Design for graceful degradation, not perfection.
  7. Measure first, optimize what's actually slow (the p99, not the mean).
Final principle: good architecture isn't about handling load — it's about handling load and failure and change, simultaneously, while a 3 a.m. on-call engineer can still understand it.

Step-by-step: how the request finds a healthy pod

  1. API Gateway → OrderService: the gateway resolves the logical name orders.svc.cluster.local via service discovery and gets a list of pod IPs.
  2. Load balancing: a Kubernetes Service (kube-proxy / iptables / IPVS) or an Envoy sidecar picks one of the 5 pods using round-robin, least-connection, or EWMA.
  3. Health check: only pods passing readiness probes are in the pool. A failing pod is removed in seconds; in-flight requests retry to a healthy peer.
  4. OrderService pod-2 → UserService: the same dance — discovery + LB picks one of the 8 user pods.
  5. Persistence: any pod writes to the same shared database / Redis. Pods are interchangeable workers — the data lives outside.

The non-negotiables that make this work

1. Statelessness

No pod stores anything that another pod would need. Sessions, carts, idempotency tokens → Redis. Files → S3. If pod-2 dies mid-request, the client retries and pod-4 serves it identically.

2. Service Discovery

Pods are ephemeral — IPs change every deploy. Discovery (k8s DNS, Consul, Eureka) maintains a live registry. Clients (or sidecars) refresh this list every few seconds.

3. Client-side or sidecar LB

For internal service-to-service traffic, an L7 LB on every hop adds latency. Modern stacks use either a client library (gRPC's built-in LB) or an Envoy sidecar per pod that does discovery + LB + retries locally.

4. Shared State Externalized

Database, cache, message broker — all clustered themselves and reachable from every pod. The 5 order pods all talk to the same Postgres primary (or the same Redis cluster). Consistency lives in the data layer, not the service layer.

5. Idempotency & Retries

If pod-2 crashes after writing the order but before responding, the client retries — and pod-4 might process it. The order endpoint must use an idempotency key (Idempotency-Key header) so the same order isn't created twice.

6. Connection Pooling

Each order pod opens a small pool of HTTP/2 or gRPC connections to user pods (multiplexed) and to the DB (via PgBouncer). Without pooling, 5 × 8 = 40 fresh TCP+TLS handshakes per second is wasted CPU.

Concurrency & data correctness across pods

Multiple order pods can process two requests for the same user at the same time. That's where things break if you're not careful:

  • Race conditions: two pods both check "stock > 0" and both decrement. Fix with DB-level row locks (SELECT … FOR UPDATE), optimistic concurrency (version column + WHERE version = ?), or an atomic operation (UPDATE … SET stock = stock - 1 WHERE stock > 0).
  • Distributed locks when DB-level isn't enough: Redis Redlock or ZooKeeper for "only one pod processes this user's checkout at a time".
  • Event ordering: publish to Kafka with partition_key = user_id so all events for one user land on the same partition and are consumed in order — even though producers are spread across pods.
  • Cache coherence: when pod-3 updates a user's profile, pod-7 may still serve a stale cache entry. Mitigate with short TTLs, cache-invalidation events on Kafka, or a write-through cache.

Sticky sessions — when (and when not) to use them

Most stateless services don't need stickiness. You need it when:

  • WebSocket / SSE connections — the user's open socket lives on one pod.
  • In-memory rate-limit counters per user (cheaper than Redis at extreme scale).
  • Cache locality: route the same user_id to the same pod via consistent hashing so its in-process LRU stays warm.

Use a hash of user_id at the LB; falls back to another pod if the chosen one is unhealthy. Avoid IP-based stickiness — corporate NATs collapse thousands of users into one IP.

Auto-scaling: how 5 pods become 50

  • Kubernetes HPA watches CPU / memory / custom metrics (RPS, p95 latency, queue depth). Scales pod replicas in/out.
  • If no node has room, the Cluster Autoscaler spins up a new VM.
  • New pods register with discovery → traffic flows to them within ~10–30 seconds.
  • Scale on the right signal: CPU is fine for compute-bound services; for I/O-bound ones, scale on RPS or active-connection count instead.
  • PodDisruptionBudgets prevent the autoscaler/k8s upgrade from killing too many pods at once.

Putting it together

The mental model: a multi-node service is a pool of identical, disposable workers behind a load balancer. Any pod can serve any request. State lives in the data layer. Coordination happens through the database, the cache, or the message broker — never in pod memory. Add or kill pods freely; the system absorbs it because no single pod is special.

What changes as you scale further

  • 10 pods → 100 pods: connection counts to the DB explode → introduce PgBouncer / RDS Proxy.
  • 100 pods → 1000 pods: service mesh becomes worth the complexity (mTLS, traffic shifting, observability uniformity).
  • One region → multi-region: each region runs its own pod fleet + cache + DB replica. Use latency-based DNS to pick a region; reconcile writes via async replication or CRDTs.
  • Hot tenant problem: one customer floods 30% of traffic to one pod via consistent hashing → split into "cells" (shuffle sharding) so noisy neighbors only affect a fraction of pods.
