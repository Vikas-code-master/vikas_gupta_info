1 API works locally but slow only in production ↑ top

Likely causes
  • Network latency — prod hits real DBs, caches and downstream services over the network; locally they're in-memory or on localhost.
  • Data volume — large prod tables expose missing indexes, full table scans and N+1 queries invisible on tiny local data.
  • Cold starts, TLS handshakes, DNS resolution, connection pool sizing.
  • Resource limits — prod containers have CPU/memory caps and noisy neighbors.
Check first: APM traces / p99 latency breakdown to see where the time goes (DB vs downstream vs app), then DB slow-query logs and pool metrics.

2 Kafka consumers run normally, but lag keeps increasing ↑ top

Likely causes
  • Throughput mismatch — producers publish faster than consumers process (heavy DB writes / downstream calls per message).
  • Partition skew / too few partitions — parallelism is capped by partition count; adding consumers beyond that does nothing.
  • A slow/poison message blocking a partition; long GC pauses; rebalancing storms.
Fix
  • Increase partitions + consumer instances, batch/async processing, profile per-message cost.
  • Tune max.poll.records / max.poll.interval.ms and watch for frequent rebalances.

3 DB connections suddenly exhausted at peak traffic ↑ top

Likely causes
  • Pool too small for concurrency, or connections leaked (never closed/returned).
  • Slow queries hold connections longer under load → pool drains.
  • No acquire/query timeouts, so requests pile up waiting.
  • Too many app instances × pool size > DB max_connections.
Fix
  • Right-size pool, add acquisition + query timeouts, fix leaks (try-with-resources).
  • Add a connection proxy (PgBouncer), optimize slow queries.

4 Autoscaling adds pods, but response time still climbs ↑ top

Likely causes
  • The bottleneck is shared & downstream, not the app — database, cache or a third-party service. More pods = more pressure on the same DB → worse.
  • Lock contention / row locks, a single-threaded resource, or a queue behind everything.
  • Scaling on the wrong metric (CPU) when the real limit is DB connections or I/O.
Fix: find the shared bottleneck and scale/optimize that — add caching, read replicas, or backpressure.

5 Retry logic creates duplicate payment transactions ↑ top

Likely causes
  • Retries on a non-idempotent operation — the request succeeded but the response/ack was lost, so the client retries and charges again.
Fix
  • Idempotency keys — client-generated unique key per transaction; server dedupes.
  • Deduplication at the payment gateway; only retry on safe/known-failed states.
  • Exactly-once semantics or an outbox pattern. Make it idempotent before retrying.

6 Scheduled job runs multiple times after scaling ↑ top

Likely causes
  • Each instance runs its own scheduler — N replicas = N executions.
Fix
  • Distributed lock (Redis/ZooKeeper) or leader election.
  • Clustered scheduler (Quartz JDBC store, ShedLock) or a single external scheduler / central job queue that dispatches once.

7 One slow downstream service affects the whole platform ↑ top

Likely causes
  • Resource exhaustion cascade — callers block on the slow service, holding threads/connections → pools fill → healthy requests starve.
  • No isolation between dependencies.
Fix
  • Always set timeouts; add circuit breakers.
  • Bulkheads — isolate thread pools per dependency; fallbacks / graceful degradation; go async where possible.

8 APIs randomly return 500s, but infra looks healthy ↑ top

Likely causes
  • App-level — unhandled exceptions, race conditions, intermittent NPEs, serialization errors on edge-case data.
  • Dependency hiccups — transient DB/cache/downstream timeouts surfacing as 500.
  • Connection/thread pool exhaustion under bursts; expired credentials/tokens; one bad pod.
Fix: structured error logging with correlation IDs, distributed tracing, error rate per pod/endpoint, and read the actual stack traces — healthy infra ≠ healthy app logic.

9 Health checks pass, but users still face failures ↑ top

Likely causes
  • Health check is too shallow — liveness returns 200 without checking dependencies. App is "up" but DB/cache/downstream is broken.
Fix
  • Meaningful readiness/dependency checks (verify DB, cache, critical downstreams).
  • Distinguish liveness vs readiness; add synthetic/end-to-end probes on real user paths.

10 Cache helps initially, then returns stale data ↑ top

Likely causes
  • Invalidation gap — data updated in DB but cache not evicted/updated; TTL too long.
  • Write path bypasses cache; multiple writers; replication lag.
Fix
  • Proper strategy: write-through / write-behind / cache-aside with explicit eviction.
  • Sensible TTLs, version/timestamp keys, event-driven invalidation.
"There are only two hard things in computer science: cache invalidation and naming things."

11 Logs exist everywhere, but cross-service debugging is hard ↑ top

Likely causes
  • No correlation/trace ID propagated across services — can't stitch one request's journey.
  • Inconsistent formats, no centralization.
Fix
  • Distributed tracing (OpenTelemetry/Jaeger); propagate a trace ID through all calls.
  • Centralized structured (JSON) logging (ELK/Loki); link logs ↔ traces.

12 JVM memory slowly increases after every deployment ↑ top

Likely causes
  • Memory leak — objects retained unintentionally: static collections, unbounded caches, thread-locals, unclosed resources, listener registrations.
  • ClassLoader leaks — classic on redeploys; the old classloader can't be GC'd.
Fix
  • Heap dump + analyze (MAT); watch old-gen growth across GC cycles.
  • Bound caches, fix leaked resources. Per-deploy specifically → suspect classloader/static state retention.

13 Works in staging but fails behind the production gateway ↑ top

Likely causes
  • Gateway-layer differences — stricter timeouts, request/response size limits, header stripping/rewriting, path rewriting, auth/JWT validation, rate limiting, CORS, TLS/mTLS.
  • Buffering, body-size caps, or WAF rules blocking traffic.
Fix: diff gateway config staging vs prod, read the gateway's own logs (it emits its own 4xx/5xx/timeouts), verify timeouts, limits, headers, auth and routing.

14 Thread pools exhausted even though CPU is stable ↑ top

Likely causes
  • Threads are blocked on I/O, not computing — waiting on slow DB/downstream calls or locks.
  • Low CPU + exhausted pool = classic blocking-wait signature.
Fix
  • Add timeouts on all blocking calls; size pools correctly.
  • Use async/non-blocking I/O, bulkhead isolation; take a thread dump to see what they're blocked on.

15 Circuit breakers configured, but cascading failures persist ↑ top

Likely causes
  • Misconfigured thresholds/timeouts — breaker opens too late; threads pile up before it trips.
  • No timeout behind the breaker → calls hang regardless.
  • Shared thread pool (no bulkhead) so one dependency starves all; breaker on the wrong layer; retries amplifying load.
Fix
  • Pair breakers with aggressive timeouts + bulkheads.
  • Tune failure thresholds / half-open behavior, limit retries, add fallbacks. Breakers alone don't help without timeouts and isolation.
