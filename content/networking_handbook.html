1. What is a Network?

A computer network is two or more devices connected together so they can exchange data. The connection may be a copper wire, an optical fiber, a radio wave, or even a satellite link. The shared rules that let those devices understand each other are called protocols.

The internet is the largest network in human history — billions of devices speaking a small handful of universal protocols (IP, TCP, UDP, DNS, HTTP), tied together by routers operated by tens of thousands of independent organizations.

The fundamental question: "How does a packet of bits get from my laptop in Bangalore to a server in Oregon, in roughly a tenth of a second, reliably, securely, and cheaply?" Every chapter below answers a piece of that question.

4. OSI Model — The 7 Layers

The OSI (Open Systems Interconnection) reference model is the canonical mental model for understanding networking. Real protocols don't always follow it strictly — but every networking conversation uses its vocabulary.

What each layer adds

  • L1 Physical — defines voltages, modulation, connectors. Talks bits.
  • L2 Data Link — frames, MAC addresses, local error detection. Talks frames.
  • L3 Network — IP addressing and routing across networks. Talks packets.
  • L4 Transport — end-to-end reliability, ports, multiplexing. Talks segments.
  • L5 Session — establishing, managing, terminating sessions.
  • L6 Presentation — encoding, compression, encryption.
  • L7 Application — what the user actually interacts with.

6. Encapsulation — How Layers Stack

As data travels down the sender's stack, each layer wraps the previous payload in its own header (and sometimes trailer). On the receiving side, each layer unwraps its header and hands the payload up. This is called encapsulation.

7. Physical Layer — The Wire

Physically transmitting bits. Modulation, signaling, connectors.

Common media

  • Twisted pair (Cat5e/Cat6/Cat6a/Cat7): standard Ethernet copper, 100 m max, up to 10 Gbps.
  • Coaxial cable: cable internet (DOCSIS), broadcast TV.
  • Fiber optic: single-mode (long haul, lasers, kilometers) and multi-mode (short, LEDs). Speeds 10/40/100/400 Gbps and beyond.
  • Wireless: Wi-Fi (2.4/5/6 GHz), cellular (4G/5G), Bluetooth, satellite.

8. Data Link Layer & Ethernet

Responsible for moving frames over a single physical link. Adds source/destination MAC addresses, error detection (FCS), and resolves contention on shared media (CSMA/CD historically).

Ethernet frame format (IEEE 802.3)

The EtherType field signals what's inside — 0x0800 = IPv4, 0x86DD = IPv6, 0x0806 = ARP, 0x8100 = VLAN tag.

9. MAC Addresses & ARP

A MAC address is a 48-bit identifier burned into a network interface card, written as six hex bytes: 3c:22:fb:1a:9b:7e. The first 24 bits identify the manufacturer (OUI); the last 24 are unique per device.

ARP (Address Resolution Protocol) maps an IP address to a MAC address on the local network. When you want to send to 192.168.1.10, your computer broadcasts: "Who has 192.168.1.10? Tell 192.168.1.5." The owner replies with its MAC. Your OS caches the answer.

$ arp -a
? (192.168.1.1) at b8:27:eb:1f:2d:9a [ether] on en0
? (192.168.1.10) at 3c:22:fb:1a:9b:7e [ether] on en0
ARP spoofing is a classic attack: poison the cache so traffic to a victim's IP gets sent to the attacker's MAC instead. Mitigations: dynamic ARP inspection on switches, static ARP entries, 802.1X.

10. Switches, Hubs & Bridges

DeviceLayerBehavior
HubL1Repeats every bit out every port. Obsolete.
BridgeL2Old 2-port switch separating collision domains.
SwitchL2 (some L3)Learns MAC↔port mapping, forwards frames only to the right port.
RouterL3Forwards packets between different IP networks.
L3 SwitchL2 + L3Hardware-accelerated routing inside a campus.

How a switch learns

  1. Frame arrives on port 3 with source MAC aa:bb:cc:00:00:01.
  2. Switch records aa:bb:cc:00:00:01 → port 3 in its CAM table.
  3. If destination MAC is known, forward only out that port. If unknown, flood to all ports except the source.

11. VLANs & Trunking

A Virtual LAN logically partitions a single physical switch into multiple isolated LANs. Tags are added to Ethernet frames per the 802.1Q standard (4-byte tag with a 12-bit VLAN ID).

Trunk port: carries multiple VLANs between switches, all tagged. Access port: connects an end device, usually one untagged VLAN.

Used to separate departments (HR, Engineering, Guest Wi-Fi) on the same physical infrastructure, isolate security zones, and reduce broadcast domains.

13. IPv4 Addressing

An IPv4 address is 32 bits, written as four dotted decimal octets: 192.168.1.10. There are 2³² ≈ 4.3 billion addresses — exhausted years ago, hence NAT and IPv6.

IPv4 packet header

Key fields: TTL prevents infinite loops; Protocol identifies the L4 (6=TCP, 17=UDP, 1=ICMP); Fragmentation supports MTU mismatches.

Address classes (legacy) and special ranges

  • Loopback: 127.0.0.0/8 — the local machine (127.0.0.1).
  • Private (RFC 1918): 10/8, 172.16/12, 192.168/16 — not routable on the internet.
  • Link-local: 169.254/16 — auto-assigned when DHCP fails.
  • CGNAT: 100.64/10 — used by ISPs for carrier-grade NAT.
  • Multicast: 224.0.0.0/4.
  • Broadcast: 255.255.255.255.

14. Subnetting & CIDR

CIDR (Classless Inter-Domain Routing) notation: 192.168.1.0/24. The number after / is how many leading bits identify the network. The remaining bits identify hosts.

CIDRSubnet maskHostsCommon use

Worked example — splitting 192.168.1.0/24 into 4 subnets

192.168.1.0/26    →  hosts .1   – .62      (network .0,   broadcast .63)
192.168.1.64/26   →  hosts .65  – .126     (network .64,  broadcast .127)
192.168.1.128/26  →  hosts .129 – .190     (network .128, broadcast .191)
192.168.1.192/26  →  hosts .193 – .254     (network .192, broadcast .255)
Tip: remember the powers of 2: /24 = 256 addresses, /25 = 128, /26 = 64, /27 = 32, /28 = 16, /29 = 8, /30 = 4. Subtract 2 for usable hosts (network + broadcast).

15. IPv6 — The Future (and Present)

128-bit addresses written as 8 colon-separated hex groups: 2001:0db8:85a3:0000:0000:8a2e:0370:7334. Consecutive zero groups can be compressed to :: (once per address).

2¹²⁸ ≈ 3.4 × 10³⁸ — enough to give every grain of sand on Earth a planet's worth of addresses.

Why IPv6 matters

  • No NAT needed — every device gets a globally routable address.
  • Stateless address autoconfiguration (SLAAC) — devices generate their own addresses from a router-advertised prefix.
  • Built-in IPsec support, simpler header, no header checksum (reduced router load).
  • Mandatory in modern mobile networks (T-Mobile, Reliance Jio).

Common prefixes

  • ::1/128 — loopback
  • fe80::/10 — link-local
  • fc00::/7 — unique local (private)
  • 2000::/3 — global unicast

16. NAT & Private IPs

Network Address Translation lets many devices on a private network share a single public IP. Your home router rewrites outgoing packets to use its public IP and remembers the mapping so it can return inbound replies to the right device.

Types: Static NAT (1:1), Dynamic NAT (pool), PAT/NAPT (port-mapped — what your home router does, also called "overloading").

Tradeoffs: conserves IPv4 space and adds a security boundary, but breaks end-to-end principles, complicates VoIP / WebRTC, and requires NAT traversal techniques (STUN, TURN, ICE, hole punching).

17. Routing & Routers

A router reads the destination IP of an incoming packet and forwards it out the appropriate interface based on its routing table. Each entry maps a destination prefix to a next-hop and outgoing interface.

$ ip route
default via 192.168.1.1 dev wlan0   proto dhcp metric 600
10.0.0.0/8 via 192.168.1.1 dev wlan0
192.168.1.0/24 dev wlan0 proto kernel scope link src 192.168.1.10

Longest prefix match: if multiple routes match, the most specific (longest CIDR) wins. 10.5.0.0/16 beats 10.0.0.0/8 beats 0.0.0.0/0 (the default route).

Forwarding vs Routing

  • Routing: the control-plane process of building the table (BGP, OSPF, static).
  • Forwarding: the data-plane act of moving each packet using that table — done in hardware in modern routers (ASIC).

BGP — the protocol that runs the internet

The internet is ~75,000 autonomous systems (ASes) — independently administered networks identified by a number (AS15169 = Google). BGP is how they exchange reachability information: "To reach prefix X, send packets through me; my path is AS-X → AS-Y → AS-Z."

BGP optimizes for policy (business contracts, peering agreements), not shortest path. A misconfigured BGP announcement can take down half the internet — see the Facebook outage of 2021, Pakistan's accidental hijacking of YouTube in 2008, Cloudflare's June 2022 outage.

19. ICMP, Ping & Traceroute

ICMP (Internet Control Message Protocol) is the diagnostics layer for IP. Messages: Echo Request/Reply (ping), Destination Unreachable, Time Exceeded, Redirect.

Ping

$ ping -c 3 google.com
PING google.com (142.250.192.78): 56 data bytes
64 bytes from 142.250.192.78: icmp_seq=0 ttl=117 time=12.4 ms
64 bytes from 142.250.192.78: icmp_seq=1 ttl=117 time=12.1 ms
64 bytes from 142.250.192.78: icmp_seq=2 ttl=117 time=11.9 ms

Traceroute

Sends packets with TTL=1, then TTL=2, etc. Each router decrements TTL by one; when it hits zero, the router sends back an ICMP Time Exceeded — and you learn its address. Repeating gives you the entire path.

$ traceroute google.com
 1  router.local       (192.168.1.1)    1.2 ms
 2  isp-gw.example.net (10.20.30.1)     8.4 ms
 3  *                                   *
 4  72.14.215.85                        12.1 ms
 5  142.251.49.43      (google peering) 11.9 ms
 6  142.250.192.78                      12.4 ms

20. TCP — The Reliable Workhorse

Transmission Control Protocol provides a reliable, ordered, byte-stream connection between two endpoints. It hides packet loss, reordering, duplication, and rate mismatches from applications.

The 3-way handshake

TCP segment header

  • Source & Destination ports (16 bits each)
  • Sequence number & Acknowledgement number (32 bits each)
  • Flags: SYN, ACK, FIN, RST, PSH, URG
  • Window size — flow control
  • Checksum

How TCP delivers reliability

  • Sequence numbers let the receiver detect loss, duplicates, and reorder.
  • Acknowledgements tell the sender what's been received.
  • Retransmission after a timeout (RTO) or fast retransmit (3 duplicate ACKs).
  • Sliding window for flow control — receiver advertises how much buffer it has.
  • Congestion control — sender keeps a congestion window (cwnd) that grows slowly and shrinks on loss. Algorithms: Reno, CUBIC (default in Linux/macOS), BBR (Google, used by YouTube).

Connection termination — 4-way handshake

Either side sends FIN, the other ACKs, then sends its own FIN, then the first ACKs. The closer enters TIME_WAIT (~60s) to absorb late packets.

TCP states

LISTEN → SYN_SENT/SYN_RCVD → ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED (and the server-side sequence). Diagnose with ss -tan or netstat -an.

21. UDP — Send and Forget

User Datagram Protocol is connectionless and unreliable: a tiny 8-byte header (source port, destination port, length, checksum) and that's it. No handshake, no retransmission, no ordering.

Why use it? When latency matters more than reliability, or when the application implements its own reliability:

  • DNS queries (small, retry is cheap)
  • Video / voice streaming (a lost frame is better than a stalled stream)
  • Online games (timely is more important than complete)
  • QUIC (custom reliability + loss recovery on top of UDP)
  • NTP, SNMP, DHCP

22. QUIC — TCP's Modern Replacement

Google designed QUIC to fix TCP's pain points without waiting decades for OS kernels to update. It runs over UDP in userspace.

  • 0-RTT or 1-RTT handshakes (vs TCP+TLS taking 2-3 RTTs).
  • Multiple streams in one connection — no head-of-line blocking like HTTP/2 over TCP.
  • Connection migration — switch from Wi-Fi to 5G without dropping the connection (uses connection IDs, not IP+port).
  • Always encrypted with TLS 1.3.

QUIC is the transport for HTTP/3, used by Google, Facebook, Cloudflare, and increasingly the rest of the web.

23. Ports & Sockets — Deep Dive

A port is a 16-bit unsigned integer (0–65535) that identifies a specific service or connection endpoint on a host. Ports give one IP address the ability to host dozens of simultaneous services. The same IP can serve HTTP on 80, SSH on 22, and a database on 5432 at the same time.

Sockets — the OS abstraction

A socket is the kernel's handle for a network endpoint. The pairing (protocol, IP, port) identifies a listening socket. A connected socket is uniquely identified by the 5-tuple:

( protocol, src IP, src port, dst IP, dst port )
  TCP        10.0.0.5  54321     142.250.192.78  443

This 5-tuple uniqueness is why one server on port 443 can handle millions of concurrent clients — each connection is distinguished by the client's IP+port.

The socket life cycle (TCP server)

socket()   → create a socket (returns a file descriptor)
bind()     → associate with an IP and port
listen()   → mark as passive, set accept queue depth
accept()   → block until a client connects; returns new connected socket
read()/write() or recv()/send() → exchange bytes
close()   → tear down

Key socket options (production gotchas)

  • SO_REUSEADDR — lets a server rebind to a port still in TIME_WAIT after a crash.
  • SO_REUSEPORT — multiple processes can bind to the same port; kernel load-balances across them (NGINX uses this).
  • TCP_NODELAY — disables Nagle's algorithm; critical for low-latency small-message protocols.
  • SO_KEEPALIVE — periodic probes to detect dead peers.
  • SO_LINGER — control behavior of close() when unsent data exists.

Ephemeral port exhaustion

Classic production bug: a load balancer opens millions of short-lived TCP connections to a backend. Each connection uses one source port. With ~28,000 available and TIME_WAIT holding them for 60 seconds, you cap at ~460 connections/sec per (src IP → dst IP:port). Fix: connection pooling, longer-lived connections, more source IPs, or tcp_tw_reuse.

Port forwarding & publishing

  • DNAT / port forwarding: router rewrites destination public:80 → 192.168.1.10:80 to expose an internal server.
  • UPnP / NAT-PMP: client asks router to auto-open a port (gaming consoles, BitTorrent).
  • SSH tunneling: ssh -L 8080:db.internal:5432 bastion maps local 8080 to a remote internal port — classic DBA trick.
  • Docker port publishing: -p 8080:80 maps host 8080 → container 80 via iptables DNAT.
  • Kubernetes NodePort: exposes a service on every node at a port in 30000–32767.

Pop quiz: why does DNS listen on both UDP and TCP port 53? UDP for normal queries (fast, fits in one datagram); TCP for zone transfers and responses over 512/4096 bytes (DNSSEC records are large). Modern DNS over TLS uses 853, DNS over HTTPS rides regular 443.

Port knocking & security through obscurity

A host keeps a service "invisible" (port filtered); only when a client sends connection attempts to a specific sequence of closed ports does the firewall open the real one. Defence in depth, not primary security.

24. DNS — The Internet's Phone Book

Domain Name System translates human-friendly names (google.com) into IP addresses (142.250.192.78). It's a hierarchical, distributed database queried over UDP/53 (TCP/53 for large responses, DoT 853, DoH 443).

Resolution flow

  1. Browser checks its cache, then OS cache (nscd, mDNS, /etc/hosts).
  2. Asks the configured recursive resolver (your ISP, 1.1.1.1, 8.8.8.8).
  3. Resolver checks its cache. If miss, queries a root server (one of 13 logical anycast clusters, A–M).
  4. Root replies with TLD nameservers (e.g., .com).
  5. Resolver queries the TLD server, gets authoritative nameservers for google.com.
  6. Resolver queries the authoritative server, gets the A record.
  7. Resolver caches and returns to the client (TTL applies).

Modern DNS

  • DNSSEC — cryptographic signatures on DNS records.
  • DoT (DNS over TLS, 853) and DoH (DNS over HTTPS, 443) — encrypted queries to prevent ISP snooping.
  • EDNS Client Subnet — passes a portion of client IP to authoritatives so CDNs can return geo-optimized answers.

DNS message format

Every DNS packet — query or response — has the same structure: a 12-byte header followed by four variable-length sections.

Header flags — what they tell you

  • QR — 0 = query, 1 = response
  • OPCODE — standard query (0), inverse (1), status (2), notify (4), update (5)
  • AA — authoritative answer
  • TC — truncated (response is too big for UDP, retry over TCP)
  • RD — recursion desired (set by resolvers)
  • RA — recursion available (set by resolvers that support it)
  • AD — authenticated data (DNSSEC-validated)
  • CD — checking disabled (client handles validation)
  • RCODE — 0 NOERROR, 2 SERVFAIL, 3 NXDOMAIN, 5 REFUSED, 9 NOTAUTH

Recursive vs iterative queries

ModeUsed byBehavior
RecursiveClient → resolver"Do whatever it takes, give me the final answer."
IterativeResolver → root/TLD/authEach server returns its best referral; resolver follows the chain.
ForwarderResolver → another resolverPass-through to a bigger, upstream resolver (e.g., corporate DNS → 8.8.8.8).

Caching & TTLs

Every DNS record has a TTL (time to live) in seconds. Resolvers cache responses until TTL expires. Typical values:

  • Root / TLD delegations: ~2 days
  • A / AAAA for websites: 300–3600 s (low for flexibility, high for efficiency)
  • MX records: often 3600–86400 s
  • Before a migration: drop TTL to 60 s a day ahead, then change, then raise back.

Negative caching (RFC 2308)

"This name does not exist" is cached too — typically for the SOA's MINIMUM field (~5 minutes). That's why a newly created record can still return NXDOMAIN for a while after it's live.

The 13 root servers (and why it's really thousands)

There are 13 logical root server identities, labeled A through M (a.root-servers.net ... m.root-servers.net). 13 is the maximum that fit in a 512-byte UDP response. Each one is actually an anycast cluster of hundreds of physical servers around the world, all advertising the same IP via BGP. When you query "a root server," you hit whichever anycast instance is closest — currently over 1,500 root server instances total. That's how 13 identities serve the entire planet.

Zones, delegation & glue records

  • A zone is the slice of the DNS tree that one authoritative server administers (e.g., everything under example.com that isn't delegated out).
  • Delegation: the parent zone (.com) publishes NS records pointing to the child zone's nameservers.
  • Glue records: when a zone's nameservers live inside that zone itself (e.g. ns1.example.com for example.com), you have a chicken-and-egg problem. The parent returns "A record" glue along with the NS records to break the cycle.

#              ^TTL         ^class ^type ^data

Important flags to spot in the HEADER line:

  • qr — it's a response
  • rd — recursion desired (you asked)
  • ra — recursion available (the resolver supports it)
  • aa — authoritative (missing → this was from a cache)
  • ad — DNSSEC authenticated
  • tc — truncated (retry over TCP)

Useful dig invocations

dig google.com                   # default A lookup
dig google.com ANY +noall +answer
dig +trace google.com            # simulate the recursive walk yourself
dig @8.8.8.8 google.com          # query a specific server
dig -x 142.250.192.78            # reverse lookup
dig MX gmail.com
dig +short google.com            # just the answer
dig +dnssec example.com          # show RRSIGs
dig AXFR example.com @ns1.example.com   # zone transfer (usually refused)

DNS load balancing techniques

  • Round-robin DNS: multiple A records for one name; resolvers rotate. Simple but no health checking.
  • Weighted records: managed DNS providers return different records with different probabilities.
  • Latency-based / Geo-based: Route 53, Cloud DNS return the IP of the closest region (using EDNS Client Subnet).
  • Failover: provider health-checks backends, returns only healthy ones.
  • Anycast: the real magic — one IP, many PoPs, BGP picks the closest. Used by root servers, 1.1.1.1, CDNs.

Split-horizon / split-brain DNS

Same name, different answers depending on who's asking. intranet.company.com resolves to 10.0.0.50 when queried from inside the corporate network, and to a VPN login page externally. Implemented with DNS "views" (BIND) or separate authoritative zones for internal vs external.

Dynamic DNS (DDNS)

Clients (home routers, IoT devices, DHCP servers) update their A record dynamically when their IP changes. Protocols: RFC 2136 UPDATE messages with TSIG authentication. Services: No-IP, DuckDNS, Cloudflare API.

DNSSEC in one paragraph

Each zone signs its records with a private key (ZSK, Zone Signing Key). The public key is published as a DNSKEY record and its hash as a DS record in the parent zone. The parent's DS is signed by the parent's ZSK, which itself has a DS in its parent, forming a chain up to the root's Key Signing Key — which is the famous "internet root key" managed by IANA in signed ceremonies. Resolvers with a copy of the root KSK can validate any signed domain end-to-end. DNSSEC prevents tampering but not eavesdropping — for that you need DoT/DoH/DoQ.

26. HTTP/2 & HTTP/3

HTTP/1.1 (1997) — what we're improving on

  • Text-based, one request per connection at a time (head-of-line blocking).
  • Browsers open 6+ parallel TCP connections per origin to compensate.
  • Header repetition is wasteful.

HTTP/2 (2015)

  • Binary framing, single TCP connection.
  • Multiplexing: many streams share one connection.
  • HPACK header compression.
  • Server push (mostly disabled by major browsers).
  • Still suffers TCP head-of-line blocking — one lost packet stalls every stream.

HTTP/3 (2022)

  • Runs over QUIC (UDP), eliminating TCP HoL blocking.
  • Faster connection setup (0-RTT resumption).
  • Connection migration across networks.
  • Mandatory encryption.

27. TLS & Certificates

Transport Layer Security (the successor to SSL) provides three things over an unreliable network: confidentiality (encryption), integrity (MAC), and authentication (certificates).

TLS 1.3 handshake (1-RTT)

How certificates work

  1. Server presents its X.509 certificate with its public key, signed by a Certificate Authority.
  2. Client validates the certificate chain up to a root CA in its trust store.
  3. Client & server use (EC)DHE to derive a shared symmetric key with forward secrecy.
  4. All subsequent traffic is encrypted with AES-GCM or ChaCha20-Poly1305.

Certificate Transparency logs publish every issued cert; browsers refuse certs not in CT logs. Let's Encrypt made TLS free with the ACME protocol.

SNI (Server Name Indication) is the TLS extension that tells the server which hostname you want — necessary so one IP can host many HTTPS sites. ECH (Encrypted Client Hello) hides SNI from network observers.

29. WebSocket & Server-Sent Events

WebSocket upgrades an HTTP connection (via Upgrade: websocket) into a persistent, full-duplex byte channel. Used for chat, live trading, multiplayer games, collaborative editors.

Server-Sent Events (SSE) is a one-way stream from server to client over plain HTTP (text/event-stream). Simpler than WebSocket — used by ChatGPT-style streaming responses.

WebRTC for browser-to-browser real-time audio, video, and data — uses STUN/TURN/ICE for NAT traversal and DTLS-SRTP for encryption.

30. DHCP — Getting an IP Automatically

Dynamic Host Configuration Protocol (UDP 67/68) hands out IP addresses, gateway, DNS servers, and other config when you join a network. Four-step DORA exchange:

  1. Discover: client broadcasts "any DHCP servers out there?"
  2. Offer: server replies with a candidate IP and lease.
  3. Request: client formally requests that IP.
  4. Acknowledge: server confirms.

Lease has a TTL (e.g., 24h); client renews at half the lease time.

30a. More Protocols — Deep Dive

Beyond HTTP and DNS, the internet runs on dozens of protocols — each solving a specific problem. Here's what you'll meet in production.

SSH — Secure Shell (TCP 22)

Encrypted, authenticated remote login, replacement for Telnet/rlogin. Carries interactive shells, file transfer (SCP, SFTP), and arbitrary TCP tunnels.

  • Handshake: version exchange → key exchange (Curve25519 / ECDH) → server host-key verification → client auth (password, public key, keyboard-interactive, GSSAPI).
  • Host key pinning: first connection stores the server's fingerprint in ~/.ssh/known_hosts; mismatch on later connections is a loud warning (MITM detection).
  • Channels: one SSH connection can carry many multiplexed channels — shell, exec, X11, port forwards.
# Generate a keypair
ssh-keygen -t ed25519 -C "you@example.com"

# Copy public key to remote host
ssh-copy-id user@server

# Local port forward — expose a remote DB on your laptop
ssh -L 5432:db.internal:5432 bastion.example.com

# Remote port forward — expose local service to the server
ssh -R 8080:localhost:3000 jumphost

# Dynamic SOCKS proxy — all traffic through the remote host
ssh -D 1080 bastion.example.com

FTP / FTPS / SFTP (TCP 21 / 990 / 22)

ProtocolSecurityNotes
FTPNoneTwo channels (control 21, data 20). Passive vs active mode.
FTPSTLSFTP wrapped in TLS. Explicit (AUTH TLS on 21) or implicit (990).
SFTPSSHA subsystem of SSH, port 22. Totally different protocol from FTP. Recommended default.
SCPSSHSimple copy over SSH. Deprecated in OpenSSH 9+ in favor of SFTP.

Telnet (TCP 23) — Historical & Debugging

The original remote-login protocol (1969). Plaintext — a sniffer reads passwords. Do not use for login. Still handy to poke at text protocols:

$ telnet smtp.example.com 25
220 smtp.example.com ESMTP Postfix
HELO test
250 smtp.example.com

Modern equivalent: nc (netcat) or openssl s_client for TLS.

NTP — Network Time Protocol (UDP 123)

Synchronizes clocks across a network to within milliseconds over the internet, microseconds on a LAN. Without NTP, TLS certificates fail (clock skew), Kerberos tickets get rejected, distributed databases corrupt.

  • Stratum 0: reference clocks (GPS, atomic, cesium). Not network-accessible.
  • Stratum 1: directly attached to stratum 0. These are the primary public NTP servers.
  • Stratum 2–15: each level synchronizes from the one above.
  • Stratum 16: unsynchronized (error).

NTP corrects for network latency by sending four timestamps (T1 client send, T2 server recv, T3 server send, T4 client recv) and computing offset + round-trip. Public pool: pool.ntp.org. Linux client: chronyd (replacing older ntpd).

PTP (Precision Time Protocol, IEEE 1588) goes further — sub-microsecond via hardware timestamping. Used in high-frequency trading and telecom.

SNMP — Simple Network Management Protocol (UDP 161 / 162)

Used to monitor and configure network gear (routers, switches, UPS units, printers). Agent on the device exposes a tree of values (MIB) indexed by OIDs (object identifiers).

  • GET / GETNEXT / GETBULK — read values (sysUpTime, ifInOctets).
  • SET — modify config.
  • TRAP / INFORM — agent pushes an unsolicited alert to the manager on port 162.
  • v1/v2c — community-string auth (basically a password in the clear). v3 adds USM auth + encryption.

Syslog (UDP 514, TCP 6514 for TLS)

De-facto standard for log forwarding. Every Unix daemon, router, and firewall speaks it. Messages have a priority (facility × severity), timestamp, hostname, and free-form text.

  • Facilities: kern, mail, auth, cron, daemon, local0–local7, etc.
  • Severities: emerg, alert, crit, err, warning, notice, info, debug.
  • Modern replacements: rsyslog, syslog-ng, journald, structured log pipelines (Fluentd, Vector, Loki).

LDAP — Lightweight Directory Access Protocol (TCP 389 / 636 TLS)

Hierarchical directory for users, groups, devices, config. Foundation of corporate identity: Active Directory, FreeIPA, OpenLDAP. Bind to authenticate, search under a base DN, filter with LDAP queries.

dn: uid=alice,ou=People,dc=example,dc=com
objectClass: inetOrgPerson
cn: Alice Smith
mail: alice@example.com
uid: alice

Kerberos (TCP/UDP 88)

Single sign-on via tickets. A client authenticates once to the KDC (Key Distribution Center), receives a TGT (Ticket-Granting Ticket), and then requests service tickets for individual servers. Clock skew > 5 minutes kills Kerberos. Backbone of Active Directory auth.

RADIUS (UDP 1812/1813) & TACACS+ (TCP 49)

Centralized authentication, authorization, accounting (AAA) for network equipment. VPNs, Wi-Fi enterprise login (WPA2-Enterprise / 802.1X), VPN concentrators, switches all forward credentials here.

MQTT — Message Queuing Telemetry Transport (TCP 1883 / 8883)

Tiny publish/subscribe protocol designed for IoT: sensors with 64 KB of RAM, flaky networks, battery-powered devices. Hierarchical topics (home/kitchen/temperature), QoS 0/1/2, last-will messages, retained messages.

Used by: AWS IoT, Azure IoT Hub, Home Assistant, Tesla cars, industrial SCADA.

AMQP — Advanced Message Queuing Protocol (TCP 5672 / 5671)

Full-featured enterprise messaging: exchanges, queues, bindings, routing keys, transactions. RabbitMQ's native protocol.

gRPC & HTTP/2 RPC (TCP 443 / 50051)

Google's RPC framework: Protocol Buffers for schema and binary encoding, HTTP/2 for transport (multiplexed streams, header compression). Four call types: unary, server-streaming, client-streaming, bidirectional. Dominant for internal microservice-to-microservice traffic.

SIP & RTP — VoIP (UDP/TCP 5060, UDP dynamic)

  • SIP (Session Initiation Protocol): signaling — who's calling, ringing, answered, hung up. Looks a lot like HTTP.
  • SDP (Session Description Protocol): carried inside SIP messages, describes codecs and port numbers.
  • RTP (Real-time Transport Protocol): actual audio/video streaming, usually UDP, with RTCP for stats.
  • SRTP: encrypted RTP (DTLS-SRTP in WebRTC).

RDP — Remote Desktop Protocol (TCP 3389)

Microsoft's pixel-streaming remote desktop. Moves screen updates, keyboard, mouse, clipboard, printing. Famous target for brute-force attacks — never expose to the internet without a VPN or gateway.

Efficient file sync that transfers only the differences between source and destination using a rolling checksum. Backbone of backup scripts, CI deployments, mirror servers.

IRC & Matrix & XMPP — Chat Protocols

  • IRC (TCP 6667): 1988. Plaintext, simple, still running (Freenode → Libera.Chat).
  • XMPP / Jabber (TCP 5222): XML-based, federated. Once used by Google Talk, Facebook Chat, WhatsApp internals.
  • Matrix (HTTPS): modern federated chat with end-to-end encryption (Olm/Megolm). Element is the reference client.

NNTP (TCP 119 / 563)

Usenet news transport, largely historical but still in use for flooding binary content across the net.

TFTP (UDP 69)

Trivial FTP. Zero-auth, zero-security, tiny. Used during PXE net-boot, router firmware upload, embedded device provisioning. Works only on the LAN.

ARP / NDP (Layer 2 protocols)

Already covered in §9 — ARP for IPv4, NDP (Neighbor Discovery Protocol, ICMPv6) for IPv6. NDP also handles router advertisements, stateless address autoconfiguration (SLAAC), and duplicate address detection.

IGMP & Multicast (IP protocol 2)

IGMP lets hosts join/leave multicast groups. Routers use it to prune trees. Multicast is widely used inside datacenters (stock market feeds, IPTV) but rarely crosses the public internet.

VXLAN, GRE, Geneve — Tunneling

  • GRE (IP proto 47): generic encapsulation — wrap any L3 packet in another IP packet.
  • IP-in-IP, SIT, 6in4: tunneling IPv6 over IPv4 for transition.
  • VXLAN (UDP 4789): layer-2 over UDP for datacenter overlays (24-bit segment IDs → 16M VLANs).
  • Geneve (UDP 6081): extensible successor to VXLAN, used in OVN and AWS VPC.

WebRTC building blocks

  • STUN (UDP 3478): "what's my public IP?" — lets a client discover its NAT mapping.
  • TURN (UDP/TCP 3478, TLS 5349): relay server when both peers are behind symmetric NAT.
  • ICE: framework for trying every candidate path to find one that works.
  • DTLS-SRTP: encryption for the media streams.

31. CDN & Edge Networks

A Content Delivery Network caches static assets (images, JS, CSS, video, sometimes API responses) at hundreds of points of presence (PoPs) close to users. DNS resolution and BGP anycast steer users to the nearest PoP.

How it works:

  1. You configure your domain to use the CDN (CNAME or NS records).
  2. The CDN's authoritative DNS returns the IP of the nearest edge.
  3. The edge serves cached content; on cache miss, it fetches from your origin.
  4. The edge applies image optimization, compression, TLS termination, WAF rules.

Major CDNs: Cloudflare, Akamai, Fastly, AWS CloudFront, Google Cloud CDN, Bunny.net.

Edge compute: Cloudflare Workers, Fastly Compute@Edge, Vercel Edge Functions — run code at the PoP for sub-50ms global latency.

32. Load Balancers

Distribute incoming traffic across multiple backend servers for scale and fault tolerance.

TypeLayerExamples
L4 (TCP/UDP)TransportAWS NLB, Linux IPVS, HAProxy TCP mode
L7 (HTTP)ApplicationNGINX, HAProxy, AWS ALB, Envoy, Traefik
DNS-based—Route 53 latency routing, AWS Global Accelerator
Anycast IPNetworkCloudflare, Google, AWS Global Accelerator

Algorithms

  • Round-robin — simple rotation.
  • Least connections — to the server with fewest active.
  • IP hash — sticky sessions based on client IP.
  • Weighted — heavier servers get more traffic.
  • Power of two choices — pick 2 random, send to less loaded (NGINX, Envoy).
  • Consistent hashing — minimal redistribution on backend changes.

Health checks

Active (LB pings backend periodically) or passive (LB observes failures from real traffic). Bad backends are removed from the pool until they recover.

33. Proxies & Reverse Proxies

  • Forward proxy: sits in front of clients, fetches the internet on their behalf. Used for caching, content filtering, anonymization, corporate egress control.
  • Reverse proxy: sits in front of servers, accepts requests on their behalf. Used for TLS termination, load balancing, caching, request coalescing, A/B testing. NGINX, HAProxy, Envoy, Traefik.
  • Service mesh sidecar: Envoy/Linkerd injected next to every microservice for L7 routing, mTLS, observability, retries.
  • SOCKS proxy: generic byte-stream proxy (used by Tor).

35. Firewalls & Network Security

  • Stateless packet filter — accepts/rejects per packet by 5-tuple. Cheap, dumb.
  • Stateful firewall — tracks connection state; allows return traffic for established connections. Linux iptables / nftables, BSD pf.
  • NGFW (Next-gen firewall) — adds L7 inspection, IDS/IPS, app awareness, TLS interception.
  • WAF (Web Application Firewall) — protects HTTP apps from OWASP Top 10 (SQLi, XSS, etc.).
  • Cloud security groups — VPC-level stateful firewalls in AWS, GCP, Azure.

Common attacks

  • DDoS: volumetric (UDP flood, amplification via DNS/NTP/Memcached) or application (Slowloris, HTTP flood).
  • MITM: ARP spoofing, BGP hijacking, rogue Wi-Fi.
  • DNS cache poisoning (mitigated by DNSSEC, source-port randomization).
  • SYN flood (mitigated by SYN cookies).

36. SDN & Cloud Networking

Software-Defined Networking separates the control plane (deciding how to route) from the data plane (actually forwarding). A central controller programs switches via APIs (OpenFlow, P4, vendor SDKs).

Cloud building blocks

  • VPC — your own logically isolated network in the cloud. Subnets per AZ, route tables, IGW, NAT GW.
  • Security groups & NACLs — stateful + stateless filtering.
  • Peering & Transit Gateways — connect VPCs.
  • Direct Connect / ExpressRoute / Interconnect — private fiber from your DC to the cloud.
  • Service mesh — Istio, Linkerd for east-west microservice traffic.
  • Container networking — Kubernetes CNI plugins (Calico, Cilium with eBPF, Flannel).

38. Troubleshooting Playbook

  1. Layer 1: Cable plugged in? Link light on? ip link.
  2. Layer 2: Got an IP? ip addr. Can you ping the gateway?
  3. Layer 3: Default route present? ip route. Can you ping 1.1.1.1?
  4. DNS: Does name resolve? dig google.com. Try a different resolver.
  5. Layer 4: Port reachable? nc -zv host 443. Firewall? Security group?
  6. TLS: openssl s_client -connect host:443 -servername host. Cert valid?
  7. L7: curl -v https://host. Check status, headers, body.
  8. Path: mtr host — packet loss at any hop?
  9. Capture: tcpdump -nni any port 443.

The discipline: start at the bottom of the stack and work up. Don't tweak HTTP headers when your DNS is broken.

39. Latency, Throughput & Bandwidth-Delay Product

  • Latency: time for one packet to travel one way (or RTT for round trip). Bound by speed of light: ~5 ms per 1000 km in fiber.
  • Bandwidth: capacity of the link (bits per second).
  • Throughput: actually achieved data rate. Always ≤ bandwidth.
  • Jitter: variance in latency. Critical for real-time media.
  • BDP (Bandwidth-Delay Product): bandwidth × RTT. The amount of "in-flight" data needed to saturate a link. TCP receive window must be ≥ BDP, otherwise you'll be stalled waiting for ACKs.
Worked example: 1 Gbps link, 100 ms RTT. BDP = 10⁹ × 0.1 / 8 = 12.5 MB. So both endpoints need TCP windows ≥ 12.5 MB to fully use that link. This is why long-fat networks require window scaling and modern congestion control.

40. What Happens When You Type google.com and Hit Enter

The classic interview question and the most beautiful tour through the entire stack. We'll go from your fingertip to a rendered page, naming every protocol and every box involved.

1Keystroke & URL parsing — Browser parses what you typed. Is it a URL? A search query? Is the scheme HTTPS implied? Does it match a previously visited site? It also checks its HSTS preload list — for google.com, HTTPS is mandatory; HTTP requests are upgraded to HTTPS in the browser before any network call.
2Browser cache check — The browser checks its own HTTP cache, service worker cache, and disk cache. Nothing fresh enough? Continue.
3DNS resolution — The browser needs the IP for google.com. It checks, in order:
  • Browser DNS cache
  • OS resolver cache (nscd, systemd-resolved, mDNS)
  • /etc/hosts file
  • If no hit, ask the configured recursive resolver (often via DHCP — your router, or 1.1.1.1 / 8.8.8.8).
The recursive resolver (if it has no cache) walks: root servers (.) → .com TLD servers → google.com authoritative nameservers (ns1.google.com etc.) → returns the A record (e.g. 142.250.192.78) and AAAA (IPv6).
4ARP (or IPv6 NDP) — Now your OS has a destination IP, but to leave the LAN it must send the packet to your default gateway (your home router). It looks up the gateway IP in the ARP cache. If unknown, it broadcasts "Who has 192.168.1.1?"; the router replies with its MAC.
5Packet leaves your machine — The OS constructs:
  • Ethernet frame, dst MAC = router, src MAC = your NIC
  • IP packet, dst IP = Google's IP, src IP = your private 192.168.x.x
  • TCP SYN segment, dst port 443, src port = some ephemeral port
The frame goes over Wi-Fi (802.11) or Ethernet to your router.
6NAT at the home router — The router rewrites the source IP from your private address to its public IP, and the source port to a free port. It records the mapping so it can route the reply back to you. The packet is then forwarded to your ISP via the WAN interface.
7The internet — many hops — Your packet travels through your ISP's access network, into their core network, and out via BGP-selected routes. It hops through dozens of routers (each decrementing TTL), peering or transit links, sometimes undersea fiber. Many large sites use Anycast IP: the same IP is advertised by hundreds of PoPs worldwide; BGP routes you to the geographically/network-topologically nearest one.
8Edge server / CDN PoP — Your packet arrives at the nearest Google edge — often within 30 ms of you. A load balancer here (Maglev / Google Front End) accepts the connection on Anycast IP 142.250.192.78:443.
9TCP 3-way handshake — SYN → SYN+ACK → ACK. Took one round trip (the RTT). Now you have a TCP connection. (For HTTP/3 you'd skip TCP entirely — UDP + QUIC.)
10TLS 1.3 handshake — Client sends ClientHello with supported ciphers, key share, and SNI = google.com. Server replies with ServerHello, its certificate chain, and Finished. Both derive a shared symmetric key via ECDHE. One more RTT (or zero if resuming with PSK / 0-RTT). All subsequent application data is encrypted with AES-GCM or ChaCha20-Poly1305.
11HTTP request — Inside the encrypted tunnel: 
GET / HTTP/2
:authority: google.com
:scheme: https
:method: GET
user-agent: Mozilla/5.0 ...
accept-encoding: gzip, br, zstd
cookie: NID=...
Sent as one stream over an HTTP/2 (or HTTP/3) connection.
12Edge processing — Google's frontend may serve a cached response, route to a regional datacenter, evaluate experiments, log telemetry, and apply rate limiting / abuse detection. For a full search, it forwards to query-serving infrastructure (a fan-out across thousands of leaves indexing a slice of the web).
13Backend work — The query is parsed, rewritten, and dispatched to many backends in parallel: web index, knowledge graph, ads, news, images, spell-check, personalization. Results are scored and merged. Total budget: ~200 ms.
14HTTP response — The frontend builds the HTML response: 
HTTP/2 200
content-type: text/html; charset=UTF-8
content-encoding: br
cache-control: private, max-age=0
strict-transport-security: max-age=31536000
content-security-policy: ...
set-cookie: NID=...; Secure; HttpOnly

<!DOCTYPE html>...
Compressed with Brotli, encrypted by TLS, segmented by TCP, packetized by IP, framed by Ethernet, sent back across the same path (or a different one — IP is connectionless).
15Browser receives bytes — TCP reassembles segments in order. TLS decrypts. HTTP/2 decodes the stream. The browser starts streaming HTML parsing immediately — it doesn't wait for the full document.
16Subresource fetching — As the parser encounters <link>, <script>, <img>, the preload scanner kicks in and fires off parallel requests for CSS, JS, fonts, images. Each may go to a different origin (CDN, static asset host). HTTP/2 multiplexes them on existing connections; HTTP/3 starts new QUIC streams. DNS, TCP, TLS again for any new origin (warmed by preconnect hints).
17Render pipeline — DOM tree built from HTML. CSSOM tree built from stylesheets. Together they form the render tree. Layout computes positions and sizes. Paint generates display lists. Compositor uploads layers to GPU. JavaScript may modify the DOM, triggering reflows.
18Page is interactive — Scripts run, event handlers attach, service workers activate, the page becomes responsive. From keystroke to first paint can be under 100 ms on a fast network — a tour through every protocol we just covered, hidden behind a single search box.
The big picture: a single page load touches DNS, ARP, Ethernet, IP, TCP (or UDP+QUIC), TLS, HTTP, HTML/CSS/JS, and dozens of independent organizations' routers. The internet works because each layer does one thing well and trusts the others to do their part — exactly what the OSI model promised in 1984.
