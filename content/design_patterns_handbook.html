1. Distributed Transaction Patterns

When a single business operation spans multiple services or databases, atomicity must be coordinated. These patterns provide different tradeoffs between consistency, availability, and complexity.

1.1 Two-Phase Commit (2PC) — Deep Dive

Strong consistency Blocking SPOF (coordinator) Synchronous XA / JTA

What it is

2PC is an atomic commitment protocol introduced by Jim Gray in the late 1970s. A central transaction coordinator ensures that a transaction spanning multiple participants (resource managers, databases, queues) either commits everywhere or aborts everywhere — never a mix.

It is the textbook way to achieve ACID semantics across distributed resources. It is also the textbook example of why distributed transactions are hard.

The cast of characters

RoleResponsibility
Transaction Manager (TM) / CoordinatorDrives the protocol. Owns the global transaction ID (GTID/XID). Maintains a durable log of decisions.
Resource Manager (RM) / ParticipantA database, message queue, or other transactional resource that holds the actual data and locks. Persists its prepared state.
ApplicationInitiates the transaction, performs reads/writes through RMs, asks the TM to commit.

The two phases — step by step

Phase 1 — Prepare (a.k.a. Voting)

  1. Coordinator writes BEGIN_2PC(xid) to its log.
  2. Coordinator sends PREPARE(xid) to every participant.
  3. Each participant:
    • Acquires and holds all locks needed for the transaction.
    • Writes a PREPARE record and an undo/redo log to stable storage (write-ahead log) — must survive a crash.
    • Replies VOTE_YES (prepared, ready to commit) or VOTE_NO (cannot commit — typically due to constraint violation, deadlock, or resource exhaustion).
  4. A participant that has voted YES is in the prepared/in-doubt state — it has surrendered the right to unilaterally abort. It must wait for the coordinator's verdict.

Phase 2 — Commit (a.k.a. Decision)

  1. Coordinator collects all votes:
    • If all YES → write COMMIT to its log (this is the "point of no return"), then send COMMIT to all participants.
    • If any NO or any timeout → write ABORT, then send ABORT to all participants.
  2. Each participant applies the decision, releases locks, writes END to its log, and acks.
  3. After all acks, the coordinator writes FORGET(xid) and may purge the log entry.

State machine of a participant

Failure scenarios — the heart of 2PC

A. Participant fails before voting

Coordinator times out waiting for the vote → presumes NO → broadcasts ABORT to everyone. Safe.

B. Participant fails after voting YES (in PREPARED state)

On reboot, the participant reads its log, sees it is in PREPARED state, and asks the coordinator for the verdict. It cannot decide alone. Locks must be re-acquired or remain held in shadow form.

C. Coordinator fails before logging COMMIT

On reboot, the coordinator finds no COMMIT record → defaults to ABORT and informs all participants. Safe.

D. Coordinator fails AFTER logging COMMIT but before sending it to all

On reboot, the coordinator finds the COMMIT record and resends to everyone (idempotent). Safe because the log is the source of truth.

E. Coordinator fails after participants are PREPARED, before any decision is logged

This is the catastrophic case. Participants are in-doubt. They cannot commit (because they don't know the outcome) and cannot abort (because some peers may have committed if the coordinator logged COMMIT but crashed before fsync). Locks remain held — possibly for hours — until the coordinator recovers. Other transactions touching those rows are blocked. This is what people mean when they say "2PC is a blocking protocol."

Why blocking matters in production: a hung 2PC transaction with 30s lock timeouts can cascade into thread pool exhaustion, replication lag, and full system outage within minutes. Many shops add aggressive timeouts and "heuristic decisions" — letting a participant guess after a long enough wait. Heuristics violate atomicity and are tracked separately as anomalies.

F. Participants can also ask each other ("cooperative termination")

An optional optimization: in-doubt participants can poll their peers. If any peer is in COMMITTED or ABORTED, that's the verdict. If all peers are also in-doubt, blocking continues.

Logging requirements

Both coordinator and participants must use a write-ahead log with fsync:

RecordWhereWhen written
BEGIN_2PC(xid, participants)CoordinatorBefore sending PREPARE
PREPARE(xid, undo, redo)ParticipantBefore voting YES
COMMIT(xid) / ABORT(xid)CoordinatorAfter collecting votes — point of no return
COMMITTED(xid) / ABORTED(xid)ParticipantAfter applying decision
FORGET(xid)CoordinatorAfter all participants ack

Variants & optimizations

  • Presumed Abort — coordinator does not log ABORT records. If a participant asks about a forgotten xid, the answer is "abort." Saves disk writes on the much more common abort path. (Default in many XA implementations.)
  • Presumed Commit — symmetric: coordinator doesn't log COMMIT for an xid once forgotten. Better when commits dominate, but requires more state during execution.
  • Read-Only Optimization — a participant that only read can vote READ_ONLY; the coordinator excludes it from Phase 2 entirely.
  • One-phase commit — when only one participant is involved, skip Phase 1 (just commit normally).
  • Last-Resource Commit (Last Resource Gambit) — only one non-XA resource is allowed, processed last; reduces real 2PC overhead in mixed environments.

Industry standards & implementations

  • X/Open XA — the de-facto API standard for 2PC: xa_start, xa_end, xa_prepare, xa_commit, xa_rollback, xa_recover. Supported by Oracle, DB2, SQL Server, PostgreSQL, MySQL (InnoDB), IBM MQ, ActiveMQ, etc.
  • JTA / JTS — Java's API on top of XA. Implemented by Atomikos, Narayana (JBoss), Bitronix, and the EE app servers.
  • MS DTC — Microsoft Distributed Transaction Coordinator (Windows ecosystem).
  • PostgreSQL — supports prepared transactions via PREPARE TRANSACTION / COMMIT PREPARED / ROLLBACK PREPARED.

Pseudocode (coordinator)

function commit_2pc(xid, participants):
    log("BEGIN_2PC", xid, participants); fsync()

    # Phase 1
    votes = parallel_send(participants, "PREPARE", xid, timeout=T1)
    if any vote != YES or any timeout:
        log("ABORT", xid); fsync()
        broadcast(participants, "ABORT", xid)
        return ABORTED

    # Phase 2
    log("COMMIT", xid); fsync()              # ← point of no return
    while not all_acked:
        broadcast(remaining, "COMMIT", xid, timeout=T2)
    log("FORGET", xid)
    return COMMITTED

function on_recovery():
    for xid in log:
        if log.has("COMMIT", xid): broadcast(participants, "COMMIT", xid)
        elif log.has("ABORT",  xid): broadcast(participants, "ABORT",  xid)
        else:                        broadcast(participants, "ABORT",  xid)   # presumed abort

2PC vs 3PC vs Paxos Commit

Property2PC3PCPaxos Commit
Round trips232 (but to a Paxos quorum)

Network-partition safe?NoNoYes
Used in practice?Yes (XA)RarelySpanner, F1

Pros

  • True ACID across resources
  • Standardized (XA, JTA)
  • Battle-tested in legacy enterprise stacks
  • Easy mental model — same semantics as a single DB transaction

Cons

  • Blocking on coordinator failure; long lock holds
  • Synchronous, chatty — kills throughput at scale
  • Coordinator is a SPOF (mitigated by HA but never eliminated)
  • Ill-suited to WAN, cross-cloud, or microservice topologies
  • Heuristic decisions break atomicity
  • Many modern stacks (NoSQL, message brokers) lack XA support
Use 2PC when: all participants are XA-aware databases inside a single datacenter, transaction durations are short (<100ms), failure rates are low, and the operational team has experience tuning timeouts and recovering in-doubt transactions.

Avoid 2PC when: participants are microservices over the WAN, durations are long, partitions are common, or one participant doesn't speak XA. Use the Saga pattern instead.

1.2 Three-Phase Commit (3PC)

Non-blocking Network-partition unsafe Theoretical

Adds a pre-commit phase between prepare and commit so participants can recover without an indefinite block.

sequenceDiagram participant C as Coordinator participant P as Participant C->>P: canCommit? P-->>C: YES C->>P: preCommit P-->>C: ack C->>P: doCommit P-->>C: ack

Rarely used in practice — adds latency and still fails under network partitions.

1.3 Saga Pattern — Deep Dive

Microservice-native Eventually consistent No isolation (ACD only) Compensations required Long-lived

Origin & definition

The Saga pattern was introduced by Hector Garcia-Molina & Kenneth Salem (1987) as a way to handle long-lived transactions (LLTs) without holding locks for hours. Their key insight: if you can break an LLT into a sequence of smaller, individually-committed transactions, and provide a way to semantically undo each one, you avoid the locking pathologies of long 2PC.

In modern microservices, sagas are the dominant pattern for any business operation that spans multiple service boundaries.

Formal structure

Saga S = T1 → T2 → T3 → ... → Tn

  with compensations: C1, C2, ..., C(n-1)

  outcomes:
    success path:  T1, T2, ..., Tn        ← all committed
    failure at Tk: T1..Tk-1, Ck-1, ..., C1 ← rolled back semantically

ACD instead of ACID

Sagas guarantee Atomicity (semantically), Consistency, and Durability — but NOT Isolation. This is the central tradeoff. Other transactions can see the partial state of an in-progress saga.

Property2PCSaga
AtomicityYes (real)Yes (semantic, via compensations)
ConsistencyYesEventual
IsolationYesNo — countermeasures needed
DurabilityYesYes (each step is locally durable)
Lock durationWhole transactionPer local transaction only

The two coordination styles

A. Choreography — event-driven, peer-to-peer

No central coordinator. Each service listens for events, performs its local transaction, and emits new events that trigger the next step.

sequenceDiagram participant O as Order Svc participant I as Inventory participant P as Payment participant S as Shipping O->>I: OrderCreated (event) I->>P: InventoryReserved (event) P->>S: PaymentCharged (event) S->>O: OrderShipped (event) Note over O,S: Failure path P--xP: card declined P->>I: PaymentFailed (event) I->>O: InventoryReleased (event) O->>O: mark order CANCELLED

Choreography Pros

  • Loose coupling — no central dependency
  • Easy to add new participants
  • Natural fit for event-driven systems

Choreography Cons

  • Hard to follow the flow — "where's the saga?"
  • Cyclic dependencies via events are easy to introduce
  • Adding a new step requires editing multiple services
  • Difficult to debug, test, and observe

B. Orchestration — central state machine

A dedicated orchestrator (Saga Execution Coordinator, SEC) tells each participant what to do next. The orchestrator persists its state and drives forward and compensation flows.

sequenceDiagram participant App participant Or as Orchestrator participant I as Inventory participant P as Payment participant S as Shipping App->>Or: place order Or->>I: reserve(items) I-->>Or: reserved ✓ Or->>P: charge(amount) P-->>Or: declined ✗ Note over Or: rollback path Or->>I: release(reservation) I-->>Or: released ✓ Or-->>App: order failed (payment declined)

Orchestration Pros

  • Workflow is explicit — readable, testable
  • Easy to add steps, retries, deadlines, parallel branches
  • Centralized observability and audit
  • Better error handling and idempotency control

Orchestration Cons

  • Risk of orchestrator becoming a "god service"
  • Tighter coupling to the workflow engine
  • Single point of operational concern (HA needed)
Rule of thumb: 2-3 services and a simple flow → choreography is fine. 4+ services, conditional branches, parallel steps, human approvals, or long-running waits → use orchestration with a workflow engine (Temporal, Camunda, AWS Step Functions).

Compensating transactions — the hard part

A compensating transaction semantically undoes a previous step. It is rarely a literal rollback. Examples:

Forward step (Tn)Compensation (Cn)Notes
Reserve inventoryRelease reservationEasy — reverse the increment
Charge customerIssue refundNot a real undo — it's a new credit transaction. May take days for ACH.
Send confirmation emailSend "ignore previous email" emailCannot retract — best you can do is apologize
Print shipping labelCancel label, request returnExternal cost involved
Provision EC2 instanceTerminate itCloud bill already incurred for the minute
Key insight: compensations are forward-only business actions, not technical rollbacks. A refund is a new transaction visible to accounting. Plan compensations with the business — they are part of the domain model.

Three types of saga transactions

  • Compensatable transactions — can be semantically undone via a compensation. Most steps in early stages of a saga.
  • Pivot transaction — the "go/no-go" point. Once it commits, the saga must succeed (or use forward retry, not backward compensation). Example: actually charging the credit card after a successful authorization.
  • Retriable transactions — steps after the pivot that must eventually succeed; on failure, retry forever. Example: send confirmation email, schedule shipment.

Lack of isolation — countermeasures

Because intermediate states are visible, anomalies can occur. The saga literature names them and supplies countermeasures (Richardson, Microservices Patterns):

AnomalyWhat goes wrong
Lost updatesSaga A overwrites a value Saga B set during its window
Dirty readsSaga reads uncommitted (will-be-compensated) state
Fuzzy / non-repeatable readsRe-reading the same entity returns a different value mid-saga

Countermeasures

  1. Semantic Lock — Add a flag like order.status = PENDING. Other transactions check the flag and either wait, fail fast, or treat the entity as locked. Released by the saga's last step or compensation.
  2. Commutative Updates — Design operations to commute (e.g., increment/decrement, set add/remove) so order doesn't matter. Refund + charge of the same amount cancels out regardless of ordering.
  3. Pessimistic View — Reorder steps so the most "dangerous" updates happen last and the riskiest ones first. Reduces the window in which an anomaly can be observed.
  4. Reread Value — Before updating, re-read with optimistic concurrency control (version/etag). Abort if the entity changed under you.
  5. Version File — Record the operations applied to an entity. When an out-of-order operation arrives, replay correctly.
  6. By Value — Choose execution path based on business risk: low-risk transactions can use eventual consistency; high-risk transactions might use stronger guarantees (e.g., 2PC for >$10k payments).

Failure recovery — backward, forward, mixed

  • Backward recovery — Run compensations to roll back. Used before the pivot.
  • Forward recovery — Keep retrying until success. Used after the pivot. Requires the operation to be idempotent and the failure to be transient.
  • Mixed recovery — Real systems combine: retry with backoff up to N times, then compensate. After the pivot, only forward recovery is allowed.

Idempotency is mandatory

Every step and every compensation must be idempotent — both because messaging is at-least-once and because a saga may retry. Common techniques:

  • Idempotency key per saga step (Stripe API style)
  • Inbox table: INSERT (msg_id) ... ON CONFLICT DO NOTHING
  • Upserts and conditional updates: UPDATE ... WHERE status='PENDING'
  • Natural idempotency: setting a flag to a known value

End-to-end example: hotel + flight + car booking

BookTrip Saga (orchestrated):
  1. ReserveHotel(hotelId, dates)            → C1: CancelHotelHold
  2. ReserveFlight(flightId, pax)            → C2: CancelFlightHold
  3. ReserveCar(carId, pickup, dropoff)      → C3: CancelCarHold
  4. CapturePayment(amount)        ← PIVOT
  5. ConfirmHotel(hotelId)                   (retriable)
  6. ConfirmFlight(flightId)                 (retriable)
  7. ConfirmCar(carId)                       (retriable)
  8. SendItineraryEmail(userId)              (retriable)

  Failure scenarios:
  • Flight unavailable at step 2  → run C1 (cancel hotel hold), notify user
  • Card declined at step 4       → run C3, C2, C1 in reverse; notify user
  • Hotel confirm flaky at step 5 → retry forever with backoff; do NOT undo payment
  • User cancels after step 7     → not a saga failure — that's a separate "CancelTrip" saga

Implementation patterns

Saga as a state machine

states: PENDING → HOTEL_RESERVED → FLIGHT_RESERVED → CAR_RESERVED
                 → PAYMENT_CAPTURED → CONFIRMED
        (failure) → COMPENSATING_CAR → COMPENSATING_FLIGHT → COMPENSATING_HOTEL → ABORTED

events: HotelReserved, FlightReserved, CarReserved, PaymentFailed, ...

transitions: defined declaratively in the orchestrator

Saga log (orchestrator persistence)

Every state change is written to a durable log before the next command is sent. On crash, the orchestrator reads its log and resumes from the last persisted state. This is the saga equivalent of the 2PC coordinator log — but without the blocking semantics.

Combine with the Outbox pattern

Saga commands often need to be sent reliably alongside DB writes. The Transactional Outbox ensures the orchestrator's state change and the outgoing command are atomic.

Common mistakes & anti-patterns

  • Treating compensations as DB rollbacks. They are forward business actions visible in audit trails.
  • Forgetting idempotency. Retries will happen. Without idempotency keys, you'll double-charge customers.
  • Choreography for >5 steps. The flow becomes invisible spaghetti. Switch to orchestration.
  • No timeout per step. A hung saga step holds semantic locks indefinitely.
  • Not handling "compensation failed" cases. What if releasing inventory fails? Need an alert + manual intervention runbook.
  • Pivot in the wrong place. Putting the irreversible step too early forces forward-only recovery for everything that follows — minor failures become operational nightmares.

Pros

  • No long-held locks; per-step latency only
  • Works across services, clouds, async boundaries
  • Tolerates partial failures gracefully
  • Each step independently durable and observable
  • Scales horizontally; no SPOF for participants

Cons

  • No isolation — anomalies need countermeasures
  • Compensation logic is real domain work
  • Requires idempotency everywhere
  • Eventual consistency window is user-visible
  • Debugging requires distributed tracing + saga log
Saga vs 2PC — quick decision:
  • Single DC, all XA-aware databases, short transactions → 2PC
  • Microservices, async boundaries, long-running flows, multi-cloud → Saga
  • Mix: use 2PC within a service (DB + outbox), use Saga between services.

1.4 Transactional Outbox

Reliable event publishing At-least-once Solves dual-write

Solves the dual-write problem — atomically updating a database and publishing a message. The trick: make the message a database row.

Pros

  • Atomic w.r.t. business write
  • Reliable publishing
  • Replayable

Cons

  • At-least-once → consumers must dedup
  • Slight publish lag
  • Outbox table needs cleanup

1.5 Inbox Pattern

The consumer-side counterpart to the outbox. Stores the message ID in an inbox table within the same transaction as state changes — duplicates are detected and ignored.

BEGIN;
  INSERT INTO inbox (msg_id) VALUES ('evt-123');  -- fails on duplicate
  UPDATE inventory SET qty = qty - 1 WHERE sku = 'X';
COMMIT;

2.7 Vector Clocks & CRDTs

Vector Clock: each node maintains a counter; events carry the full vector. Lets us detect causality and concurrent updates.

Node A: [A:1, B:0]   →   [A:2, B:0]
Node B:                       [A:1, B:1]   ← concurrent with [A:2, B:0] (siblings)

CRDTs (Conflict-free Replicated Data Types): data structures that converge automatically — counters (G-Counter, PN-Counter), sets (G-Set, OR-Set), maps, sequences. Used in Riak, Redis Enterprise, Figma, Google Docs.

4. Reliability & Resilience Patterns

4.1 Circuit Breaker

Wrap remote calls in a state machine. After N failures, the breaker opens and fast-fails — protecting the downstream service from being hammered while it recovers.

// Resilience4j-style
CircuitBreaker cb = CircuitBreaker.of("payment", config);
return cb.executeSupplier(() -> paymentService.charge(amount));

4.2 Retry with Exponential Backoff & Jitter

delay = min(MAX, BASE * 2^attempt) + random_jitter(0, JITTER_CAP)

4.3 Bulkhead

Isolate resources per dependency so one slow service can't drain the entire thread pool.

4.4 Rate Limiting Algorithms

Token Bucket

Tokens refill at rate R, capped at B. Each request consumes a token. Allows bursts up to B.

Leaky Bucket

Fixed-rate output. Excess requests are queued or dropped.

4.5 Other Resilience Tools

  • Timeout — never call without one. The most-violated rule.
  • Load Shedding — drop low-priority requests under heavy load.
  • Backpressure — downstream signals upstream to slow down (reactive streams, Kafka consumer lag).
  • Idempotency Key — client supplies unique key per request; server returns cached result on retry (Stripe API).
  • Failover & Fallback — automatic switch to standby; degraded-but-useful response.
  • Health Checks — liveness, readiness, startup probes.
  • Heartbeat — periodic "alive" signal for failure detection.

7.3 Write-Through vs Write-Behind vs Write-Around

Write-Through

Write to cache and DB synchronously. Always fresh; slower writes.

Write-Behind

Write to cache; cache flushes to DB async. Fast; risk of loss on crash.

Write-Around

Write to DB only. Cache filled on read. Good for write-heavy with infrequent reads.

  • Index Tables — secondary tables for alternative lookup keys (NoSQL).
  • Soft Delete — deleted_at column instead of removing rows.

10. Coordination & Consensus

10.1 Leader Election

Multiple nodes elect a single leader to coordinate. The leader holds a lease and renews it via heartbeats.

sequenceDiagram participant N1 as Node 1 participant N2 as Node 2 participant N3 as Node 3 participant Z as etcd / ZooKeeper N1->>Z: try to acquire lease Z-->>N1: granted (you are leader) N2->>Z: try to acquire Z-->>N2: denied (N1 holds it) loop heartbeat N1->>Z: renew lease end Note over N1: N1 crashes N2->>Z: try acquire (lease expired) Z-->>N2: granted (new leader)

10.2 Raft Consensus (simplified)

Used by etcd, Consul, CockroachDB. Decomposes consensus into leader election + log replication + safety.

10.3 Distributed Lock

Mutex across nodes. Always use fencing tokens — a monotonically increasing number that the lock holder presents to downstream services. If a stale lock holder tries to write with an old token, the storage layer rejects it.

1. Acquire lock → token=42
2. Long pause (GC pause, network)
3. Lock expires; another client gets token=43, writes
4. Original wakes up, tries to write with token=42 → REJECTED ✓

10.5 Consensus Algorithms — quick reference

  • Paxos — original; theoretically beautiful, notoriously hard.
  • Raft — modern, easier to understand and implement.
  • ZAB — ZooKeeper Atomic Broadcast.
  • PBFT / BFT — Byzantine Fault Tolerance (handles malicious nodes).
  • Gossip protocols — eventually consistent membership/state (Cassandra, Consul).
Two-Generals Problem: reliable communication over an unreliable channel is impossible. We work around this with timeouts, retries, and idempotency — never with theoretical perfection.

12.4 SLI / SLO / SLA / Error Budget

  • SLI — what you measure (e.g., p99 latency).
  • SLO — internal target (p99 < 200ms 99.9% of the time).
  • SLA — external contract with consequences for breach.
  • Error budget = 100% − SLO. Burn it down → freeze deploys, focus on reliability.

13. Security Patterns

13.1 Defense in Depth

Multiple, overlapping layers — so one failure doesn't compromise the whole system.

13.2 Zero Trust

Never trust — always verify. No implicit trust based on network location. Every request: identity + device posture + policy.

13.3 OAuth 2.0 / OIDC Flow (simplified)

sequenceDiagram participant U as User participant C as Client App participant A as Auth Server participant R as Resource API U->>C: click login C->>A: redirect (auth code req) A->>U: login + consent U-->>A: credentials A-->>C: auth code C->>A: exchange code (+ secret) A-->>C: access token + ID token C->>R: API call with Bearer token R-->>C: data

13.4 Other Security Patterns

  • Principle of Least Privilege — grant minimum permissions.
  • mTLS — both sides present certs; common inside service meshes.
  • Token-based auth — JWTs (stateless) or opaque tokens (revocable).
  • Secrets Management — Vault, AWS Secrets Manager. Never in git.
  • WAF + Rate Limiting — guard against L7 abuse.
  • Encryption at rest & in transit — KMS-managed keys, TLS.
  • Input validation & output encoding — the cure for injection & XSS.
